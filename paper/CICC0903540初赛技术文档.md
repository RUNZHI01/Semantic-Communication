## 摘要

本文面向弱网条件下的视觉信息回传任务，构建了一套部署在飞腾平台上的多核语义通信系统。系统采用发送端-接收端分离架构：上位机负责图像语义编码、信道扰动模拟与任务下发，飞腾派 Linux 侧负责语义特征解码与图像重建；在需要展示安全控制能力时，再启用 OpenAMP/RTOS 控制面，承担 admission、heartbeat、safe stop 与故障收口等职责。我们将数据面与控制面明确分离，以保证工程叙事与真实实现一致，不把 OpenAMP 误写为推理加速来源。

在当前统一口径下，`4-core Linux performance mode` 用于描述飞腾多核上的高性能重建能力：healthy-board 条件下，TVM serial current 为 `231.522 ms/image`，big.LITTLE pipeline current 为 `134.617 ms/image`，同轮吞吐提升 `56.077%`，PyTorch default reference 为 `484.183 ms/image`。`3-core Linux + RTOS demo mode` 用于描述 OpenAMP 安全控制面的板级闭环：`STATUS_REQ/RESP`、`JOB_REQ/JOB_ACK`、`HEARTBEAT_ACK`、`SAFE_STOP`、`JOB_DONE` 以及 `FIT-01/02/03` 已具备真机证据。这种系统化表述使本项目从单点推理框架优化工作，转化为更符合飞腾杯要求的系统级作品。

## 1. 系统定位与总体方案

### 1.1 系统定位与双模式设计

本文将作品定位为一个面向弱网场景的飞腾多核安全语义视觉回传系统，而不是单一的 TVM/MNN 优化实验。系统的核心目标是在发送端完成图像语义编码，在飞腾侧完成语义重建，并在真实工程约束下同时给出可复验的多核性能路径与可审计的安全控制路径。

从工程结构看，系统采用发送端-接收端分离架构。上位机负责图像预处理、Encoder 推理、语义特征量化、信道扰动模拟与任务编排；飞腾派 Linux 侧负责 Generator 推理与图像重建，是系统的数据面主路径；在需要展示异构协同与安全控制时，再启用 OpenAMP/RTOS 控制面，承担 admission、heartbeat、safe stop、fault accounting 等职责。控制面仅传输 `job_id`、`expected_sha256`、`input_shape`、`heartbeat` 等小控制消息，不承担大张量数据搬运。

为了避免模式混写，本项目对外统一采用以下两种 operating modes：

1. **`4-core Linux performance mode`**：用于性能 headline 与系统吞吐展示。该模式下飞腾 Linux 侧保持 4 核在线，当前默认引用的 healthy-board 结果为 TVM serial current `231.522 ms/image`、big.LITTLE pipeline current `134.617 ms/image`，相对同轮 serial current 吞吐提升 `56.077%`。

2. **`3-core Linux + RTOS demo mode`**：用于演示 OpenAMP 控制面与安全执行边界。该模式下通过 `remoteproc` 拉起 RTOS 控制面，会占用一个 Linux CPU，因此必须诚实表述为 `3-core Linux + RTOS`，其价值在于 admission、heartbeat、safe stop 与故障注入验证，而不是 headline performance。

> **图 1.1** 系统总体架构与双模式说明图

实验上位机使用 2023 款联想拯救者 Y7000P，配置 NVIDIA GeForce RTX 4060 Laptop GPU（8GB）和 13th Gen Intel(R) Core(TM) i7-13700H 处理器。当前工程中，上位机通过 OpenSSH 与飞腾派建立安全连接，将量化后的语义特征发送至板端并触发解码执行；飞腾派完成重建后，将结果保存到指定目录，形成弱网语义视觉回传的端到端闭环。

### 1.2 关键技术与工程实现

1. **语义通信驱动的弱网视觉回传。** 系统不直接传输原始像素，而是在发送端提取语义特征并发送紧凑 latent，再由飞腾侧重建图像。这种“先理解、后回传”的路径更适合带宽受限、链路波动和端侧算力有限的弱网场景。

2. **飞腾 Linux 数据面上的多核高性能重建。** 本文以 TVM 作为静态语义解码主线，以 MNN 作为动态尺寸备线。TVM current 路线已经形成 safe runtime、target 收敛、baseline-seeded warm-start incremental tuning、真实重建 benchmark 与 artifact SHA guard 组成的可信闭环；对外 headline 统一引用 `4-core Linux performance mode` 下的结果，不把 OpenAMP 写成性能来源。

3. **OpenAMP/RTOS 安全控制面。** 在演示模式中，OpenAMP 控制面负责 `STATUS_REQ/RESP`、`JOB_REQ/JOB_ACK`、`HEARTBEAT_ACK`、`SAFE_STOP`、`JOB_DONE` 等最小控制闭环，并通过错误 SHA、输入契约违规、watchdog timeout 等 FIT 场景验证控制路径的真实性与可审计性。该能力用于证明系统的安全执行边界，而不是替代 Linux 数据面推理。

4. **可信传输与部署治理。** 当前数据面沿用 OpenSSH 作为上位机与飞腾派之间的安全通信通道，并已在同一局域网和跨网络场景中完成验证。与此同时，artifact SHA guard、输入契约校验和板级证据归档共同保证了部署版本可追溯、异常输入可拦截、关键结果可复验。

## 2. 模型原理

### 2.1 背景和语义通信技术介绍

随着数字化和信息化时代的加速演进，图像与视频数据已成为网络流量增长的主要驱动力。截至 2025 年 4 月，全球互联网用户数量已达 58.1 亿，占全球人口的 70.7%，同比增长 2.6%；与此同时，全球数据总量预计将从 2010 年的 2 泽字节（ZB）增长至 2028 年的 394 泽字节[1][2]。这意味着在未来通信系统中，如何在带宽受限、链路波动和端侧算力有限的条件下完成有效的视觉信息回传，将成为一项持续存在的工程问题。

本项目选择语义通信路线，并不是为了单独验证某一种推理框架，而是因为弱网场景下不宜直接回传原始像素。我们在发送端先提取图像语义特征，再将紧凑 latent 传回飞腾侧执行重建，把通信压力从像素级传输转化为语义级传输[4][5]。对飞腾平台而言，这一任务天然包含两个层面：一是如何利用多核算力完成高性能重建，二是如何在真实部署中给出可信的控制与安全边界。

当前通信技术凭借先进的编解码与调制技术，已逼近香农信道容量的理论极限。与此同时，人工智能（AI）在通信技术演进中的作用日益突出[4]。**语义通信技术**（Semantic Communication）作为一种新兴的通信范式，通过理解和传输信息的语义内容，旨在实现更高效、更智能的通信方式。

在传统通信系统中，数据由源编码器进行压缩，再由信道编码器添加冗余以增强抗干扰能力；接收端执行逆过程恢复原始数据。在这种模块化结构中，收发两端不涉及对消息含义的理解，信息的隐含语义被完全忽略[4]。语义通信技术则在此基础上引入语义层，系统在发送前提取消息的语义特征，在接收时重构语义信息，而非进行单纯的比特传输。这种方式能够显著减少冗余数据的传输量，尤其在带宽受限的场景下可更有效地利用有限的信道资源[5]。这种“理解后再传输”的理念使语义通信成为下一代（6G）通信系统中极具前景的关键技术[5]。

语义通信系统的复杂性在于，发送端和接收端不仅需要完成传统通信终端的功能，还需要运行高度智能的语义提取与推断算法。发送端能够从原始消息中提取语义特征并编码传输，接收端则需要“理解”所接收到的语义并据此重建信息[4]。在本文的工程实现中，这一过程进一步被拆分为飞腾 Linux 数据面与 OpenAMP/RTOS 控制面，以便在性能、可靠性和答辩口径之间保持一致。

面向即将到来的 6G 时代，AI 被视为最核心的使能技术之一。AI 驱动的 6G 将提供自聚合、情境感知、自配置等新型能力，推动无线电技术从认知无线电向智能无线电的转变[3]。此外，6G 的目标不仅在于服务人口密集地区，还要通过低成本方案将网络覆盖扩展至偏远地区，实现全球通信公平性[3]。语义通信通过提取和传输数据的语义信息来大幅减少冗余数据量，顺应了 6G 时代极低时延的技术趋势，特别适用于增强移动宽带（eMBB）和超可靠低时延通信（URLLC）场景。同时，语义通信能够基于上下文推断和恢复信息，即使在传输过程中出现干扰或丢包，也能保证通信的有效性，契合 6G 高可靠性的技术目标。

语义通信的应用前景广阔。在物联网场景中，语义通信因其较低的无线电资源消耗和对信道噪声的相对鲁棒性，可实现精确的实时数据传输。然而，物联网设备的计算和存储资源有限，难以直接运行复杂的深度神经网络（DNN），因此如何压缩语义编解码器模型成为关键挑战。常用的压缩手段包括网络稀疏化和量化等技术[4]。本项目正是采用了经过压缩的语义通信神经网络，将其部署在算力受限的飞腾派端侧设备上，并通过 TVM、MNN 与 OpenAMP 控制面协同构成可落地的弱网语义视觉回传系统。

### 2.2 模型原理介绍

本系统所采用的模型来自文献[9]（已获第一作者授权使用），该文提出了基于生成对抗网络（GAN）的图像语义通信方案及其轻量化版本。

**生成对抗网络**（Generative Adversarial Networks, GANs）是一类基于博弈论思想的深度生成模型，由生成器网络（Generator）和判别器网络（Discriminator）两部分组成。在对抗训练过程中，Generator 学习生成逼近真实数据分布的样本，Discriminator 则学习区分真实样本与生成样本，二者交替优化，最终使 Generator 具备生成高质量图像的能力。

文献[9]综合了 DeepJSCC 和条件 GAN（cGAN）的思想，构建了一种基于 GAN 的图像语义联合信源-信道编码方案（GAN-based JSCC），其结构如下图所示。

> **图 1.2** 基于 GAN 模型的图像语义通信方案 (GAN-basedJSCC) 结构图

本部分对模型的核心原理进行介绍。在发送端，编码器网络（Encoder）将输入图像映射到高维特征空间，提取出最具代表性的语义特征向量。在原始模型中，语义特征向量 \(y\) 经过功率控制后，以模拟符号的形式直接通过物理信道发射。在接收端，部署生成器网络（Generator），根据信道输出的受噪声干扰的特征向量 \(\hat{y}\) 重建与原始图像内容接近的图像 \(\hat{x}\)。具体来说，Generator 将 \(\hat{y}\) 变换为高维图像张量：

\[
\hat{x} = G(\hat{y}) \in \mathbb{R}^{H \times W \times C}
\]

其中 \(\theta_G\) 为生成器的可学习参数，\(H, W, C\) 分别表示图像的高度、宽度和通道数。

Generator 旨在通过优化其参数 \(\theta_G\)，使得生成样本 \(\hat{x}\) 在判别器 D 的评估下被误判为真实样本的概率最大化。这一过程可形式化为最小化以下损失函数：

\[
L_G = \mathbb{E}_{\hat{x} \sim p(\hat{x})} [\log(1 - D(G(\hat{x})))]
\]

从整个图像语义通信流程来看，图像首先通过发送端神经网络转换为信息量更加密集的特征向量，然后通过物理信道发送给接收端。接收端神经网络根据特征向量重建出保持相同内容的图像，图像语义通信主要是通过神经网络的重建能力来减小中间通信资源的开销。为了使用 cGAN 模型增强 Generator 的图像生成能力，训练过程中还会引入一个判别器网络（Discriminator），输入是真实图像 (\(x\) 或者生成图像 \(\hat{x}\))，判别的同时也要输入特征向量 \(y\)，输出则是在以特征向量 \(y\) 为基础判断输入图像是真实图像的概率，输出概率越接近 1 则表示输入图像越可能是真实图像。Discriminator 网络仅在训练阶段用于进行对抗训练，在推理阶段则弃用，仅在接收端设备部署 Generator 网络用于图像重建。

具体来说，Discriminator 网络作为一个二元分类器，接收输入样本（真实样本或生成样本 \(\hat{x} = G(\hat{y})\)），并输出标量值 \(D(x, \theta_D) \in [0,1]\)，表示输入样本为真实数据的置信度，其中 \(\theta_D\) 为判别器的可学习参数。判别器的优化目标是通过最大化对真实样本和生成样本的区分能力来调整 \(\theta_D\)，其损失函数定义为：

\[
L_D = -\mathbb{E}_{x \sim p_{data}(x)}[\log D(x)] - \mathbb{E}_{z \sim p(x)}[\log(1 - D(G(x)))]
\]

传统框架仅使用 MSE 作为损失函数，容易导致重建图像出现模糊。为此，GAN-based JSCC 引入了 LPIPS（Learned Perceptual Image Patch Similarity）感知损失函数，该指标能够更好地模拟人类视觉系统的感知特性，从而在保真度和视觉质量之间取得更优平衡。本模型的训练损失由以下几部分组成：

为了保证重建图像对原始图像的保真度，考虑 MSE 损失：

\[
L_{MSE} = \mathbb{E}_{p(x)}[d(\hat{x}, x)]
\]

通过 AlexNet（一个经典的预训练模型）第 \(l\) 层的输出，分别记为 \(z^l \in \mathbb{R}^{H_l \times W_l \times C_l}\) 和 \(\hat{z}^l \in \mathbb{R}^{H_l \times W_l \times C_l}\)，通过对通道维度进行单位归一化得到。感知误差距离 \(d_{LPIPS}\) 可按如下公式计算：

\[
d_{LPIPS}(z, \hat{z}) = \sum_{l} \frac{1}{H_l W_l} \sum_{h,w} \| w_l \odot (z^l_{hw} - \hat{z}^l_{hw}) \|_2^2
\]

其中向量 \(w_l\) 表示第 \(l\) 层的权重，\(\odot\) 表示 Hadamard 乘积。LPIPS 的损失函数定义为：

\[
L_{LPIPS} = \mathbb{E}_{p(x)}[d_{LPIPS}(f_{pre}(x), f_{pre}(\hat{x}))]
\]

其中 \(f_{pre}\) 表示预训练的 AlexNet 网络。LPIPS 损失捕捉神经网络层输出的特征向量之间的距离。设 \(L_G = -\mathbb{E}_{p(x)}[\log D(G(\hat{y}), y)]\) 表示生成损失，则编码器与生成器的训练目标定义如下：

\[
L_t = \lambda L_G + \alpha L_{MSE} + \beta L_{LPIPS}
\]

其中 \(\lambda, \alpha, \beta\) 是控制各损失项权重的超参数。判别器通过损失函数 \(L_D\) 进行训练，其定义为：

\[
L_D = -\mathbb{E}_{p(\hat{x})}[\log(1 - D(G(\hat{y}), y))] - \mathbb{E}_{p(x)}[\log D(x, y)]
\]

上文所提出的模型的详细结构如图 (a) 所示。编码器主要利用三个卷积-归一化-ReLU 级联模块和两个残差块从输入图像中提取特征。生成器同样包含卷积-归一化-ReLU 级联模块和残差块。然而，残差块的引入显著增加了参数数量和计算复杂度，这给计算资源有限的设备（如移动终端和物联网设备）的部署带来了挑战。为了在诸如移动通信和物联网等场景中提高图像语义通信的实际部署潜力，采用了一种生成对抗网络（GAN）压缩策略对生成器 \(f\) 进行压缩，训练出一个更轻量级的生成器 \(\hat{f}\)，同时尽量保留重建图像的质量。

本文中所用的 GAN 压缩方法如图所示。受知识蒸馏启发，在 GAN 压缩策略中，训练学生模型去模仿教师模型的中间层，这使得即使学生网络架构比教师网络简单得多，也能高效地训练学生网络。为了训练学生网络，定义蒸馏损失如下：

\[
L_{distill} = \sum_{t=1}^{T} \| f_t(\hat{f}(\mathbf{y})) - f_t(f(\mathbf{y})) \|_2^2
\]

其中 \(T\) 为选定层数，\(\hat{\theta}\) 为学生生成器的可训练参数，\(\hat{f}_t\) 和 \(f_t\) 分别为学生和教师生成器第 \(t\) 层的中间层输出，\(f_t\) 是将学生特征映射到教师通道数的常规层。最终学生模型的训练损失函数为：

\[
L_s = \gamma L_{distill} + \lambda L_{GAN} + \alpha L_{MSE} + \beta L_{LPIPS}
\]

以上为模型的核心原理。下面介绍训练数据、信道模型以及评估指标。

该模型使用包含超过 900 万张不同尺寸图像的 **Open Images V7** 数据集。该数据集的物体类别涵盖广泛，包括人物、动物、交通工具、食品等。从数据集中选取 110,000 张图像用于训练。

信道模型为加性高斯白噪声（AWGN）信道，信噪比（SNR）定义为：

\[
\text{SNR} = 10 \log_{10} \frac{\| \mathbf{y} \|^2}{\sigma^2} \quad (\text{dB})
\]

衡量重建图像失真的常用指标为峰值信噪比（PSNR），其定义为：

\[
\text{PSNR}(\mathbf{x}, \hat{\mathbf{x}}) = 10 \log_{10} \frac{A^2}{\| \mathbf{x} - \hat{\mathbf{x}} \|^2} \quad (\text{dB})
\]

**LGJSCC 的训练分为两个阶段：**

1. **第一阶段：** 训练生成式 JSCC 作为教师模型，超参数设置为 \(\lambda = 0.15\)、\(\alpha = 0.0023\)、\(\beta = 1.0\)，编码器和生成器分别使用 2 个和 5 个残差块。采用对抗训练策略，生成器与判别器交替迭代训练。教师模型生成器的各层通道数固定为 \(c_t\)。

2. **第二阶段：** 训练 LGJSCC 学生模型，将教师模型第 \(t\) 层的 \(c_t\) 个通道压缩至 \(\hat{c}_t\) 个通道。通过修改通道配置 \(\hat{c}_t\) 实现不同压缩比。其余参数与第一阶段一致。学生判别器继承教师判别器的结构与权重，并与学生生成器交替训练。

在本系统中，我们考虑到飞腾派算力有限，仅采用了 **LGJSCC-c3** 模型。实际上在后续章节中，我们的后端优化方法可适用于以上所有模型，采取其他的模型可以进一步降低生成图片的失真率。在后续阶段，我们计划进一步优化其他压缩模型，以探索更优的性能与质量平衡。

需要说明的是，在当前实际部署中，我们使用 OpenSSH 在现有网络信道之上实现安全通信，尚未引入额外的物理信道。在后续工作中，我们计划为系统引入更贴近实际通信场景的物理信道，如短波信道（频率范围 3–30 MHz）。短波通信利用天波传播方式，可实现数千公里甚至上万公里的远距离通信，且相较于卫星通信系统，其建设和维护成本较低，适合验证语义通信在实际信道条件下的性能表现。

## 3. 系统运行

本部分重点介绍上位机与飞腾派端侧的测试脚本执行逻辑。首先进行模型加载：在 `test_sub.py` 中配置模型路径信息，随后调用 `test_model.py` 中的类函数完成加载，进入前向传播函数。

前向传播函数首先将模型设置为评估模式（`eval`），禁用梯度计算和参数更新，以适应纯推理场景；随后从指定目录读取图片数据，输入 Encoder 模型进行语义特征提取。

Encoder 推理输出的特征向量随后进行量化处理，将量化所需的各项参数计算并打包保存。量化的目的是进一步压缩语义特征张量的数据量，提高传输效率。

最后，执行 `test_sub.py` 中的 `scp_and_execute_commands` 传输函数。该函数首先配置 OpenSSH 的连接参数，需要提前在飞腾派开发板上启用 OpenSSH 服务并开放 22 端口，同时确保上位机与飞腾派之间的密钥认证已正确配置。

随后，进行 SSH 安全通道初始化：

1. 通过 `paramiko.SSHClient` 创建 SSH 客户端对象
2. 设置 `AutoAddPolicy` 策略自动信任未知主机密钥
3. 启用压缩传输（`compress=True`）减少网络带宽消耗

这里我们使用如下配置：

1. 选择 `aes128-ctr`/`aes256-ctr` 加密算法，利用 CPU 硬件加速指令提升加密效率。
2. 设置 `default_window_size=1GB` 扩大 TCP 窗口，提升吞吐量。
3. 设置 `REKEY_BYTES=1GB` 减少密钥重新协商频率，降低协议开销。

我们通过设置 SFTP 连接池架构提高传输效率：

1. 创建 `max_workers` 数量的 SFTP 连接，形成连接池复用机制
2. 设置 `CHUNK_SIZE=block_size`，将文件分块后进行传输，优化大文件传输效率。
3. 通过 `mmap.mmap` 实现内存映射读取，避免文件重复加载到内存，降低性能损耗。

在上一步中的 `block_size`，根据文件实际大小动态选择 16MB~64MB，平衡内存与效率。

并且，考虑到飞腾派上的内存大小和读写速度有限，我们使用 `BytesIO` 内存缓冲减少磁盘 I/O 瓶颈。

至此，上位机端测试代码介绍完成。飞腾派端侧设备接收到命令后，执行经 MNN 或 TVM 优化后的模型推理代码，对接收到的语义特征张量进行解码和图像重建。

## 4. 模型优化与系统运行

### 4.1 TVM 介绍

TVM（Tensor Virtual Machine）是一个开源的深度学习编译器，专注于优化和部署机器学习模型到多种硬件平台。它提供了一个端到端的编译栈，使开发者能够将高级深度学习模型转换为高效、低延迟的硬件加速代码。

> **图 4.1** TVM 优化编译器框架转换时所采取的步骤

TVM 的工作流程如下：首先从 TensorFlow、PyTorch 或 ONNX 等主流框架中导入模型，提取其计算图结构与参数。

导入的模型随后被转换为 TVM 的高层次中间表示（IR）。早期版本使用 **Relay** 作为高层 IR，最新版本引入了 **Relax** 作为其统一替代。Relay/Relax 提供了一个跨框架的统一抽象层，以计算图形式描述模型的高层语义，便于后续的跨平台优化与分析。

然后降级为**张量表达式**（TE）表示。降级是指将较高级的表示转换为较低级的表示。应用了高级优化之后，Relax 通过运行 FuseOps pass，把模型划分为许多小的子图，并将子图降级为 TE 表示。张量表达式（TE）是一种用于描述张量计算的领域特定语言（DSL）。TE 还提供了一些 schedule 原语来指定底层循环优化，例如循环切分、矢量化、并行化、循环展开和融合。

再然后使用 **auto-tuning** 模块搜索最佳 schedule。schedule 为 TE 中定义的算子或子图指定底层循环优化。auto-tuning 模块搜索最佳 schedule，并将其与 cost model 和设备上的测量值进行比较。TVM 提供了多种自动调优模块，包括：

- **AutoTVM：** 基于模板的 auto-tuning 模块。它运行搜索算法以在用户定义的模板中找到可调 knob 的最佳值。TOPI 中已经提供了常用算子的模板。
- **AutoScheduler（Ansor）：** 无模板的 auto-tuning 模块。它不需要预定义的 schedule 模板，而是通过分析计算定义自动生成搜索空间，然后在生成的搜索空间中搜索最佳 schedule。
- **MetaSchedule：** 作为新一代自动调优模块，结合了 AutoTVM 和 AutoScheduler 的优点，并在此基础上进行了改进，提供了一个更灵活、高效的调度优化方式。具体说来，有以下优势：
  - MetaSchedule 引入了更灵活的模板系统，用户可以更轻松地为不同算子和硬件目标自定义搜索空间。
  - 它整合了 AutoTVM 和 AutoScheduler 的经验，提供了一个统一的框架，使得调度优化过程更加高效。
  - MetaSchedule 采用了先进的搜索算法和改进的成本模型，提升了自动调优的效率和效果。
  - MetaSchedule 特别适用于复杂模型或特殊硬件的优化，能够在默认调度无法达到最优性能时提供显著改进。

这些自动调优模块会探索不同的循环优化策略（如切分、矢量化、并行化、展开和融合），并结合设备上的测量值和成本模型评估性能，最终为每个子图选择最佳调度配置。

> **图 4.2** TVM 端到端优化过程流程图

然后为模型编译选择最佳配置。调优后，auto-tuning 模块会生成 JSON 格式的调优记录。此步骤为每个子图选择最佳 schedule。

然后降级为**张量中间表示**（TIR，TVM 的底层中间表示）。基于调优步骤选择最佳配置后，所有 TE 子图降级为 TIR 并通过底层优化 pass 进行优化，进一步提升性能，例如循环重排、内存访问优化等。接下来，优化的 TIR 降级为硬件平台的目标编译器。这是生成可部署到生产的优化模型的最终代码生成阶段。

最后，目标编译器将优化后的代码生成为机器码。TVM 可将模型编译为可链接的共享库模块，通过轻量级 TVM 运行时以 C 语言 API 动态加载，同时也提供 Python 和 Rust 等语言的调用接口。此外，TVM 支持将运行时与模型打包在一起进行捆绑部署，简化端侧分发流程。

### 4.2 静态 TVM 的优化过程和效果

对于静态输入形状和输出形状的模型来说，由于搜索空间确定，使用 TVM 具有很好的优化效果。具体来说，TVM 通过多层次中间表示（Relay/Relax → TE → TIR）和自动调优技术（AutoTVM、AutoScheduler、MetaSchedule），来实现静态输入形状和输出形状的模型的高效优化和跨硬件部署。在本节中，我们将会具体介绍使用 MetaSchedule 对于静态形状的语义通信模型的优化过程和最终效果。对于动态形状模型，我们将会在下一节中讲解如何利用另一框架 MNN 进行优化。在后续工作中，我们将进一步研究如何利用 TVM 的最新进展对动态形状模型进行优化。

#### 4.2.1 静态 TVM 的优化过程

> **图 4.3** TVM 优化逻辑流程图

考虑到本项目解码端 latent 张量的输入形状固定为 `1×32×32×32`，且核心部署任务聚焦于 256×256 图像重建，因此 TVM 路线具备开展静态图深度编译优化的前提。本文并未将 TVM 工作理解为一次性的“模型导出 + 编译”操作，而是将其组织为“静态图规范化 - 安全运行时重构 - target 收敛 - 增量调优 - 真实重建验证”的闭环过程。相较于 baseline 阶段以“板端可跑通”为主要目标，current 阶段更强调三个问题：一是让 TVM 0.24 产物在飞腾派上可稳定导入并可重复执行；二是让编译目标真正反映 AArch64 微架构特征；三是让 benchmark 从纯 payload 调用推进到真实图像重建场景，从而把编译器优化转化为系统级收益。

**（1）宿主机侧静态 ONNX 规范化。**

ONNX 作为跨框架模型交换标准，能够为后续编译优化提供统一入口。我们首先在上位机复用原测试代码 `test_sub.py` 完成 PyTorch 解码器加载，再由 `decoder_optimizer.py` 的 `_convert_with_retry` 将计算图和参数分阶段导出为 ONNX：先生成不含参数的 `temp_partial.onnx`，再补入权重与偏置得到 `temp_full.onnx`。考虑到模型导出与图优化阶段计算开销较大，这一步放在高性能主机完成，而不是由飞腾派承担。随后使用 `onnxsim.simplify` 对 `temp_full.onnx` 做常量折叠、冗余节点消除与算子规范化，得到结构更稳定的 `temp_simp.onnx`。这一阶段的作用并不只是“完成格式转换”，更重要的是为 Relax 前端和 MetaSchedule 提供一个语义等价但搜索空间更可控的静态图起点。

**（2）安全运行时重构与兼容路径清理。**

baseline 所依赖的旧 compat runtime 只能支撑历史 artifact 的执行，而 current 产物在 TVM 0.24.dev0 路线上一度无法稳定导入。进一步定位表明，单纯重编译 `libtvm.so` 并不能解除阻塞；`gdb` 追踪显示，SIGILL 首先出现在 PyTorch `libc10.so` 的 `_mi_options_init`，其触发源是 `tvm_ffi` 初始化阶段对 torch 的 eager import 及可选 torch C DLPack hook。换言之，问题不在模型本身，而在运行时依赖装载链与飞腾派 ARMv8.0 指令约束之间存在不兼容。为此，我们在飞腾派上重建独立的 safe runtime：一方面重编译 TVM C++/FFI 组件并隔离 Python 环境，另一方面关闭非必需的 torch eager import 与 torch C DLPack hook，同时禁用 FP16/BF16、DNNL、IDEEP 等不适配路径。完成上述处理后，current 路线从“可编译但不可稳定加载”转变为“可导入、可初始化、可重复运行”的可信执行底座，后续 target 收敛与调优搜索才具备实际意义。

**（3）目标配置收敛与飞腾派硬件适配。**

在 TVM 0.24.dev0 中，target 已不再沿用旧式 CLI 字符串，而是统一使用 JSON/dict 目标对象。这意味着 target 选择既要符合新版编译接口，也要尽可能贴近飞腾派 CPU 的实际特征。基于 safe runtime，我们在相同数据库与相同 payload 路径下，对 `generic + neon`、`generic + neon + crypto + crc`、`cortex-a72 + neon` 与 `cortex-a72 + neon + crypto + crc` 开展真机比较。结果表明：在同口径短跑测试中，`generic + neon` 的中位时间为 **2502.638 ms**，明显慢于 `generic + neon + crypto + crc` 的 **2482.345 ms** 和 `cortex-a72 + neon` 的 **2481.304 ms**；在更长重复次数下，`cortex-a72 + neon + crypto + crc` 虽取得 **2482.095 ms** 的最低中位数，但方差升至 **933.920 ms²**，显著高于 `cortex-a72 + neon` 的 **70.625 ms²**。综合性能与稳定性，本文将 `{"kind":"llvm","mtriple":"aarch64-linux-gnu","mcpu":"cortex-a72","mattr":["+neon"],"num-cores":4}` 收敛为 current 路线的默认 target。该收敛过程的意义并不只是“更换参数”，而是让代码生成与调度搜索建立在更符合飞腾派微架构的硬件先验之上，从源头减少 target 失配带来的性能折损。

**（4）基于 baseline 数据库的热启动增量调优。**

完成前端转换后，我们调用 `tvm.relax.frontend.onnx.from_onnx` 将静态 ONNX 导入 Relax IR，并使用 `relax.get_pipeline` 与 MetaSchedule 组织编译优化。与 baseline 或早期 current 的 `total_trials=0` rebuild-only 路径不同，current 阶段不再满足于“沿用历史 schedule 的重建”，而是引入 baseline-seeded warm-start incremental tuning：以 baseline 已积累的 tuning database 作为 seed，在 RPC runner 上继续执行 `500` 次非零预算增量搜索。该策略一方面继承 baseline 对高价值 schedule 区域的先验知识，另一方面允许 current 在新 target 与新 runtime 条件下完成局部再学习，因而最终生成的 current artifact 不是简单重打包产物，而是面向 current 执行环境重新搜索得到的结果。

**（5）产物封存、SHA 守护与真实重建闭环。**

编译完成后，我们通过 `tvm.compile` 与 `export_library` 生成 `optimized_model.so`，并将 SHA-256 为 `1946b08e6cf20a1259fa43f9e849a06f50ae1230c08d4df7081fba1edae4c644` 的增量产物定义为可信 current artifact。为防止远端归档产物漂移导致 benchmark 失真，执行脚本会在运行前对实际加载的 `.so` 进行 SHA 校验。进一步地，我们补齐了 current 侧真实重建执行器 `current_real_reconstruction.py`、远端封装脚本 `run_remote_current_real_reconstruction.sh` 以及配套环境 `inference_real_reconstruction_compare.2026-03-11.phytium_pi.env`，使 benchmark 不再停留在 `main()` 负载级调用，而是能够完整覆盖 latent 读取、AWGN 扰动、VM 执行、图像解码与 PNG 落盘的真实链路。至此，TVM 路线形成了“编译产物身份可验证、运行路径可复现、评测口径可追溯”的完整实验闭环。

#### 4.2.2 静态 TVM 的优化效果

加载模型时，我们通过 TVM Relax VirtualMachine 将 `.so` 文件绑定到飞腾派 CPU 上。需要说明的是，本文中的 **baseline** 并不是“未优化模型”，而是项目早期已经完成首轮 TVM 部署和编译优化后的参考版本；**current** 则是在 baseline 基础上继续完成 safe runtime 重构、target 收敛、增量调优、真实重建执行链路补齐以及 artifact SHA 守护后的最新版本。因此，TVM 路线的性能评估不能被单一数字概括，而应按照“归因证据 - 公平负载对比 - 系统级结论”的顺序展开。

下面我们采用 **Places365 数据集**（256×256）进行测试。Places365 数据集由麻省理工学院（MIT）的计算机科学与人工智能实验室（CSAIL）于 2016 年创建，旨在推动场景识别和理解的研究。该数据集包含 365 种不同场景类别的高分辨率图像，总计超过 180 万张图片，为计算机视觉领域提供了丰富资源。本文重点报告以下三类结果，其口径与结论汇总如下。

> **表 4.1** TVM 三类性能对比口径与结论

| 对比口径 | 对比对象 | 关键结果 | 用途 |
|---|---|---|---|
| current-only（Scheme B） | rebuild-only current vs incremental current | **2479.246 ms** vs **152.36 ms**，提升 **93.85%**，加速 **16.272x** | 归因于 baseline-seeded warm-start incremental tuning 的贡献 |
| payload-symmetric（Scheme A） | baseline vs current | **1829.28 ms** vs **152.846 ms**，提升 **91.64%** | 验证在相同 `load + VM init + main()` 路径下 current 仍显著领先 |
| 真实端到端重建 | baseline vs current，双方各输出 **300 张 PNG** | **1830.3 ms/image** vs **255.931 ms/image**，提升 **86.02%** | 作为面向系统部署的最终结论 |

**（1）current-only 负载对比：用于隔离增量调优本身的贡献。**

在完全相同的 current 运行时与 payload 路径下，我们比较了“复用历史数据库直接重建”的 rebuild-only 版本与“以 baseline 数据库为 seed 的增量调优”版本。结果表明，rebuild-only current 的推理中位时间为 **2479.246 ms**，而增量调优后的 current 中位时间降至 **152.36 ms**，相对提升 **93.85%**，速度提升 **16.272 倍**。由于两者共享同一 safe runtime、同一输入形状和同一输出形状 `[1, 3, 256, 256]`，该差异可以较为直接地归因于 MetaSchedule 在新 target 条件下继续搜索得到的 schedule，而不是脚本包装或 benchmark 口径变化。

**（2）payload-symmetric 公平对比：用于验证 baseline 与 current 在同类负载路径下的差异。**

在 Scheme A 对比中，baseline 与 current 都通过相同的 `load_module + VM init + main()` 负载路径执行，且双方 artifact SHA 都与配置值严格匹配。该组实验中，baseline 的推理中位时间为 **1829.28 ms**，current 为 **152.846 ms**，提升 **91.64%**。这说明即使将执行框架统一到同一 payload 路径，current 仍然保持显著优势。不过需要谨慎指出，baseline artifact 的输出形状为 `[1, 3, 249, 249]`，而 current 为 `[1, 3, 256, 256]`；因此，该组实验更适合被解释为“运行路径公平后的负载级证据”，而不宜作为最终发布口径下的严格同构比较。

**（3）真实端到端重建对比：作为本文 TVM 路线的最终性能结论。**

在最新真机实验中，baseline 与 current 都完成了从 latent 输入读取、模型解码到 PNG 文件落盘的完整端到端流程，两侧各生成 **300 张重建图像**。实验结果表明：baseline 的单张图片重建中位时间为 **1830.3 ms/image**，current 为 **255.931 ms/image**，current 相对于 baseline 减少 **1574.369 ms/image**，端到端速度提升 **86.02%**；同时，current 侧实际加载产物的 SHA 与期望值 `1946b08e6cf20a1259fa43f9e849a06f50ae1230c08d4df7081fba1edae4c644` 完全一致。由于真实重建路径显式包含 latent 读取、AWGN 扰动注入、张量封装、VM 执行以及 PNG 编码写回等系统开销，其绝对数值高于纯 payload 推理时间属于预期现象。也正因为如此，本文将 **86.02% 的端到端提升** 作为 TVM 路线的最终结论，而将 **91.64% 的 payload-symmetric 对比** 视为 current 负载级优势的补充证据，将 **93.85% 的 current-only 对比** 视为增量调优贡献的归因证据。

#### 4.2.3 从 baseline 到 current 的工程优化演进

从 baseline 演进到 current，并不是简单替换了一份新的 `optimized_model.so`，而是将 TVM 路线从“历史版本可执行”升级为“面向飞腾派当前硬件、具备可归因证据链的竞赛级优化方案”。其性能提升不是由单一技巧触发，而是沿着执行前提、目标建模、搜索深度、评测口径和实验治理逐层消除瓶颈的结果。

**（1）从兼容运行到安全运行。**

baseline 依赖历史 compat runtime，能够支撑旧 artifact 的执行；但 current 所对应的 TVM 0.24 产物最初在 import 阶段即遭遇兼容性阻塞。只有先完成 safe runtime 重构，解决 FFI、可选依赖装载和板端指令兼容问题，current 路线才从“理论上可编译”转变为“在飞腾派上可重复执行”。因此，运行时重构不是附属工程，而是 current 路线成立的必要前提。

**（2）从保守 target 到收敛 target。**

早期为了优先保证“模型可跑通”，我们采用了较保守的 target 配置。这种做法虽然降低了执行风险，但也限制了 TVM 对飞腾派 CPU 特性的利用，是早期 current 性能偏低的重要来源之一。完成 safe runtime 恢复后，我们通过真机比较发现，`generic + neon` 的中位时间停留在 **2502.638 ms**，而 `cortex-a72 + neon` 可下降到 **2481.304 ms**；进一步考察长跑稳定性后，又确认 `cortex-a72 + neon + crypto + crc` 虽有更低中位数，但抖动明显更大。因此，默认 target 最终收敛为 `cortex-a72 + neon + num-cores=4`。这一阶段的贡献，在于将编译目标从“保守可执行”提升为“面向飞腾派微架构的稳定适配”，为后续增量调优提供了更有效的搜索基面。

**（3）从数据库复用到增量学习。**

rebuild-only 路线本质上是“沿用已有 schedule 的重编译”，可以验证 current 路线的可执行性，却难以产生新的调优知识。引入 baseline-seeded warm-start incremental tuning 后，系统开始在 baseline 历史数据库之上进行非零预算搜索，最终得到可信 current artifact（SHA-256：`1946b08e6cf20a1259fa43f9e849a06f50ae1230c08d4df7081fba1edae4c644`）。current-only 对比中 **2479.246 ms → 152.36 ms** 的跃迁，表明 current 的核心突破来自 schedule 再搜索，而非仅仅恢复了运行能力。

**（4）从 payload benchmark 到真实重建 benchmark。**

早期 current 结果主要来自 `load_module + VM init + main()` 的 payload 级测量，这类实验有利于快速确认 artifact 水平，但并不能覆盖真实部署中的输入读取、扰动注入和结果落盘开销。为此，我们补齐了 current 侧真实重建执行器 `current_real_reconstruction.py` 及其远端封装脚本 `run_remote_current_real_reconstruction.sh`，使 current 也能够像 baseline 一样完成“latent 读取 - 图像重建 - PNG 写出”的完整流程。由此得到的 **255.931 ms/image**，以及双方各输出 **300 张 PNG** 的结果，才构成面向系统部署的最终性能口径，而不只是编译器内部性能的 proxy。

**（5）从人工确认到 SHA 守护的实验可信性。**

在持续调试过程中，远端归档目录中的 `optimized_model.so` 曾出现漂移与覆盖风险，若不加约束，benchmark 结果就可能与目标 artifact 脱钩。为此，我们在 current 路线中加入 artifact SHA guard，对执行时实际加载的 `.so` 做显式身份校验，并将 benchmark 报告中的性能数字与具体 SHA 绑定。在最终真实重建实验中，实际加载产物 SHA 与期望值 `1946b08e6cf20a1259fa43f9e849a06f50ae1230c08d4df7081fba1edae4c644` 完全一致，这意味着 **255.931 ms/image** 的结论能够被明确追溯到唯一 artifact，而不再依赖“远端环境默认未变化”的隐含假设。

综上，baseline 到 current 的演进是一次从“历史版本可部署”走向“可解释、可复现、可量化优化”的方法论升级。也正因为运行时、target、调优、基准口径和 artifact 治理五个层面的协同推进，current 才能够在飞腾派真机的真实端到端重建场景下，相对于 baseline 取得 **86.02%** 的速度提升。

### 4.3 MNN 介绍

> **图 4.4** MNN logo

**Mobile Neural Network (MNN)** 是阿里巴巴开发的轻量级开源深度神经网络推理引擎，也是其端到端通用机器学习平台 Walle 的底层计算核心[6]。MNN 的两大核心创新在于：一是**几何计算机制**，将转换和复合算子分解为原子算子，大幅减少了在多个后端上逐一手动优化算子的工作量；二是**半自动搜索机制**，能够在运行时快速识别算子在不同后端上的最优实现方案[6]。

MNN 将底层的张量算子分为四类：

1. **原子算子**（Atomic Operators）：如加法、乘法等基本运算，是后端优化的基础单位。
2. **转换算子**（Transform Operators）：改变张量形状或元素顺序，如转置、切片等。
3. **复合算子**（Composite Operators）：可分解为原子和转换算子的组合，如卷积、池化等。
4. **控制流算子**（Control-Flow Operators）：如 `if` 和 `while`。

> **图 4.5** MNN 概述示意图

MNN 在架构上分为 **Converter**（模型转换器）和 **Interpreter**（推理解释器）两大模块。Converter 包含前端和图优化器：前端负责解析不同训练框架导出的模型（目前支持 TensorFlow、TensorFlow Lite、Caffe 和 ONNX），图优化器则通过算子融合、算子替换和数据布局调整来优化计算图。Interpreter 包含引擎和后端：引擎负责模型加载与计算图调度，后端负责在具体计算设备上的内存分配与算子实现。MNN 在引擎和后端层面集成了多种优化技术，包括 Winograd 卷积加速、Strassen 矩阵乘法、低精度计算、ARM Neon 向量化、手写汇编优化、多线程并行、内存复用以及异构计算等。

#### 4.3.1 几何计算机制（Geometric Computing）

目前，MNN 支持 61 个原子算子、45 个转换算子、16 个复合算子和 2 个控制流算子，并适配 16 个后端（如 CPU、GPU）。为减少在多个后端上优化所有算子的巨大工作量（原工作量 O(1954)），MNN 引入了**光栅算子**（raster operator），通过几何计算机制（geometric computing mechanism）将转换和复合算子分解为光栅算子和原子算子的组合。这样，只需优化原子算子和光栅算子，工作量将降至 O(1055)，减少约 46%。

光栅算子的核心功能是通过坐标变换和内存地址映射移动元素。例如，在切片操作中，MNN 通过遍历坐标并根据线性映射（由 strides 和 offsets 定义）将元素从输入张量移到输出张量。为支持此功能，MNN 引入了"region"概念，包含输入张量、坐标范围和内存地址映射。此外，MNN 还优化分解后的算子，如通过垂直合并和水平合并减少冗余操作。

**原子算子优化（Atomic Operator Optimization）：**

对原子算子（包括 raster operator），MNN 从多层面优化：

- **算法层面：** 采用 Winograd 和 Strassen 算法减少卷积和矩阵乘法的计算量。
- **指令集层面：** 利用 SIMD（如 ARM Neon、x86 AVX512）加速，优化数据布局和打包。
- **内存层面：** 通过 tiling 和 memory reordering 减少读写次数，提升内存连续性。
- **汇编层面：** 手写汇编代码，应用循环展开、软件流水线和指令重排。

#### 4.3.2 半自动搜索（Semi-Auto Search）

MNN 通过半自动搜索自动选择最优后端和算子实现。其目标是找到成本最低的后端，成本定义为所有算子在该后端上的最优实现成本之和。MNN 将此转化为约束优化问题，结合硬件约束和输入规模，优化算法参数（如 SIMD 打包大小、tile 大小等）。相比手动搜索，半自动搜索减少工作量并提高优化效果；相比 TVM 的 auto tuning，它利用手动经验缩小搜索空间，支持运行时优化。

MNN 的设计允许将模型作为常规资源文件部署，支持在 Python VM 中进行运行时优化和日常机器学习任务迭代，且不会因任务增加而显著增大移动应用包大小。这种高效、灵活的张量计算设计使 MNN 非常适合资源受限的移动和嵌入式设备。

#### 4.3.3 Session API

根据 MNN 官方文档，使用 MNN 推理时，有两个层级的抽象，分别是解释器 **Interpreter** 和会话 **Session**。Interpreter 是模型数据的持有者，负责加载和解析模型文件，并为推理提供必要的环境和会话支持。Interpreter 可以通过指定的模型文件路径（通常为 `.mnn` 文件）加载模型的结构和参数；可以解析模型文件中的计算图，理解模型的拓扑结构和权重数据，为后续推理执行做好准备。Session 通过 Interpreter 创建，是推理数据的持有者[8]。Session 可以提供执行模型前向传播的接口，可以获取输入张量、设置数据，并读取输出结果，同时还可以分配和管理推理过程中所需的计算资源，例如内存、线程等，以确保推理的高效性。

### 4.4 MNN 的动态优化过程和效果

#### 4.4.1 模型转换

由于我们拟部署并优化的模型为 PyTorch 模型，MNN 不支持 PyTorch 但支持 ONNX 格式，所以先要将 PyTorch 模型的 `.pt` 文件转换成 ONNX 格式的 `.onnx` 文件。

4.2.1 中进行了 PyTorch 模型转换成静态形状的 ONNX 模型的操作，现在需要让经 MNN 优化后的模型支持动态形状输入，则需要在 4.2.1 操作基础上做一些修改。根据 MNN 官方文档，MNN 能够运行从其他框架转换而来的模型，若原始模型（如 ONNX 或 TensorFlow 模型）定义了动态输入维度，MNN 在转换后会保留这些动态特性。

> **图 4.6** MNN 模型转换默认配置

同时在推理过程中，MNN 允许用户通过 Python API 为输入张量指定不同的形状，以适配动态尺寸的输入图像。在输入 Tensor 维度不确定或需要修改时，需要调用 `resizeTensor` 来更新维度信息，更新完所有 Tensor 的维度信息之后，需要再调用 `resizeSession` 来进行预推理，进行内存分配及复用。

> **图 4.7** resizeTensor 和 resizeSession 示例使用

在导出 ONNX 模型时，为确保模型支持动态尺寸输入，需要通过 `dynamic_axes` 参数设置动态轴，将输入和输出张量的第 2 维（高度）和第 3 维（宽度）标记为动态维度，使模型在推理时能够处理任意分辨率的输入图像。

#### 4.4.2 模型运行

在完成模型转换后，需要编写测试代码来调用模型实现系统功能，这里将主要介绍涉及 MNN 操作的部分，完整的代码放在 `mnn_006.py` 中。

Session 是通过 Interpreter 创建的推理数据的持有者，是使用 MNN 进行推理的关键接口。飞腾派 CPU 包括 2 个 1.8GHz 大核和 2 个 1.5GHz 的小核，基于多次的实验测试结果，我们选择创建 2 个 Interpreter，为每一个 Interpreter 创建一个 Session，各实现一遍模型加载。

使用程序实现待处理图片隐式特征张量的预加载，在开启模型推理前将 `.pt` 文件从 SD 卡写入 CPU 内存，减少推理计算中的数据存取开销。

通过程序使用 `getSessionInput(session)` 获取输入张量，然后通过 `interpreter.resizeTensor(input_tensor, new_shape)` 和 `interpreter.resizeSession(session)` 调整输入张量形状，为 session 分配内存，进行推理准备工作，使系统可以支持动态尺寸的图片张量输入。

使用 `runSession(session)` 来运行 session 执行模型推理，返回对应的 error code，需要根据错误码来判断后续是否成功执行模型推理。

使用 `interpreter.getSessionOutput(session)` 和 `output_tensor.getData()` 获取模型输出张量，并将其转换为 numpy 数组，之后使用 torchvision 完成图像重建。

#### 4.4.3 优化效果

我们在飞腾派上使用 300 张不同尺寸的图片对 MNN 优化后的模型进行端到端测试。实验结果表明，MNN 优化版本处理全部 300 张图片的总耗时为 **123 秒**（单张平均约 410 ms），相比优化前的并行处理版本性能提升了 **1.85 倍**。MNN 路线的核心优势在于其对动态输入尺寸的原生支持——无需将所有图片统一缩放至固定分辨率即可直接进行推理，这在实际应用场景中具有重要意义。

## 5. 总结与展望

本文围绕飞腾派端侧语义通信图像传输系统，分别基于 TVM 和 MNN 两条技术路线，完成了模型的编译优化、端侧部署与真机验证。
**在 TVM 路线方面**，本文完成的并非一次性的静态模型编译，而是一条面向飞腾派实机部署的闭环优化流程：以 safe runtime 重构解除 TVM 0.24 在板端的 import/FFI 兼容阻塞，以 JSON target 收敛完成 AArch64 微架构适配，以 baseline-seeded warm-start incremental tuning 完成 schedule 再学习，并通过真实重建执行器与 artifact SHA guard 将 benchmark 提升为可复验的系统级实验。最终得到的可信 current artifact 为 `1946b08e6cf20a1259fa43f9e849a06f50ae1230c08d4df7081fba1edae4c644`，对应的默认 target 为 `cortex-a72 + neon + num-cores=4`。

从证据层次看，TVM 的收益应被分为三层理解。第一，current-only 负载对比表明，增量调优相对 rebuild-only 将中位时间从 **2479.246 ms** 降至 **152.36 ms**，说明 current 的关键突破来自非零预算 schedule 搜索，而非简单重编译。第二，在 payload-symmetric 公平对比中，baseline 与 current 的中位时间分别为 **1829.28 ms** 和 **152.846 ms**，current 仍保持 **91.64%** 的显著优势；但由于两侧 artifact 的输出形状仍存在 `249×249` 与 `256×256` 的差异，该结果仅作为负载级佐证。第三，在最终的真实端到端重建对比中，双方均输出 **300 张 PNG**，baseline 为 **1830.3 ms/image**，current 为 **255.931 ms/image**，端到端提升 **86.02%**。因此，本文将 **86.02% 的端到端重建提升** 作为 TVM 路线的最终结论。

**在 MNN 路线方面**，本文利用 MNN 框架的几何计算机制和半自动搜索能力，实现了对动态输入形状模型的优化部署。通过模型转换、动态形状适配、双 Interpreter 并行推理以及数据预加载等工程手段，MNN 路线成功支持了不同尺寸图片的动态输入处理。在 300 张不同尺寸图片的测试中，MNN 优化后的处理耗时为 **123 秒**，相比优化前提升了 **1.85 倍**。MNN 路线与 TVM 路线形成互补：TVM 擅长对静态形状模型进行深度编译优化以获取极致性能，MNN 则在动态形状支持和轻量化部署方面具有独特优势。

本文的工作还表明，端侧模型优化不能仅以纯推理负载时间作为唯一指标。只有当模型能够在真实环境中完成隐式张量输入读取、图像重建和结果落盘，并在 artifact 身份可验证的前提下进行统一计时，编译器优化收益才能被可靠地映射为系统部署收益。因此，本文明确区分了 current-only 对比、payload-symmetric 对比与真实端到端重建对比三种口径，并以后者作为最终性能结论。

后续工作主要包括以下几个方向：

1. 继续收敛 TVM 路线中基准版本与最新版本在输出形状和历史产物上的差异，进一步提高实验比较的严格性与可解释性；
2. 进一步研究如何利用 TVM 的最新进展对动态形状模型进行优化，使系统在保持高性能的同时支持更多实际图像尺寸；
3. 深入优化 MNN 路线的推理性能，探索量化精度、算子调度等维度的进一步提升空间；
4. 在现有 OpenSSH 传输链路基础上，引入更贴近实际通信场景的物理信道模块，构建更加完整的语义通信系统验证平台；
5. 在更丰富的数据集和更复杂的信道条件下开展测试，进一步评估系统的泛化性能和部署稳定性。

总体而言，本文完成的不只是一次模型加速实验，而是基于 TVM 和 MNN 两条互补路线的系统性优化实践。相关结果表明，在飞腾派这一类资源受限的端侧平台上，只有将运行时兼容、编译目标建模、调度搜索、实验治理与真实业务链路验证协同推进，TVM 与 MNN 的框架优势才能真正转化为可复验的系统收益。这也为语义通信技术在边缘设备上的工程化落地提供了更扎实的技术依据。

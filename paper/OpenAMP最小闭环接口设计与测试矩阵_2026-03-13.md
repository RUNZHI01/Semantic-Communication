# OpenAMP 最小闭环接口设计与测试矩阵

- 对应赛事：第八届全国大学生集成电路创新创业大赛（飞腾杯）
- 文档日期：2026-03-13
- 文档性质：内部接口设计文档
- 目标：为“TVM 高性能解码 + OpenAMP 异构监护 + 安全可靠验证”提供最小可落地闭环设计

---

## 1. 文档目的

本文档用于把 OpenAMP 中阶能力从“概念方向”收敛为“可直接实现的最小闭环”。

本文只解决以下问题：

1. OpenAMP 在系统中的职责是什么
2. Linux 主核和从核分别负责什么
3. 控制消息怎么定义
4. 作业时序和状态机怎么走
5. 故障码、风险项、FIT 样例怎么落
6. 测试覆盖矩阵如何建立

本文**不**试图解决以下问题：

1. 不让 OpenAMP 承载大张量数据流
2. 不让 RTOS/Bare Metal 从核执行 TVM 推理
3. 不重写现有 trusted current TVM 执行链

---

## 2. 设计原则

### 2.1 保底原则

OpenAMP 的引入必须满足两个约束：

1. 不能破坏当前 trusted current TVM 数据面
2. 必须能够形成赛题要求的中阶与高阶闭环

### 2.2 数据面/控制面分离

数据面保持不动：

- 上位机继续做 Encoder、AWGN、SSH/SCP 传 latent
- 飞腾派 Linux 侧继续执行 trusted current artifact
- 继续沿用 safe runtime、artifact SHA guard、reconstruction benchmark

控制面通过 OpenAMP 实现：

- 作业授权
- 心跳监护
- deadline 监测
- 参数校验
- 异常拒绝
- 安全停止
- 故障上报

### 2.3 最小化实现范围

第一版 OpenAMP 最小闭环只做以下功能：

1. `JOB_REQ` 申请执行
2. `ALLOW/DENY` 执行授权
3. `HEARTBEAT` 运行中保活
4. `SAFE_STOP` 故障停止
5. `JOB_DONE` 作业收尾
6. `STATUS_REQ/STATUS_RESP` 状态查询

## 3. 当前系统中的位置

### 3.1 当前可信底座

当前可信底座保持如下事实不变：

- trusted current SHA：`6f236b07...6dc1`
- 正式 payload：`1846.9 -> 130.219 ms`
- 正式真实端到端：`1850.0 -> 230.339 ms/image`
- artifact SHA guard 已正式使用

### 3.2 最小闭环架构

```text
上位机
  ├─ Encoder / AWGN / SSH / SCP
  └─ 触发飞腾派 Linux 侧 orchestrator

飞腾派 Linux 主核
  ├─ orchestrator / guard_client
  ├─ trusted current TVM runner
  ├─ artifact SHA / input contract precheck
  └─ 通过 RPMsg 向从核发送控制消息

飞腾派 RTOS/Bare Metal 从核
  ├─ safety_guard
  ├─ job admission policy
  ├─ heartbeat watchdog
  ├─ deadline monitor
  ├─ fault counter / sticky state
  └─ SAFE_STOP / DENY / fault report
```

### 3.3 组件职责

| 组件 | 所在侧 | 职责 |
|---|---|---|
| `orchestrator` | Linux | 收作业、前置校验、驱动 TVM current 执行、维护 job 生命周期 |
| `guard_client` | Linux | RPMsg 端点、消息编码/解码、状态同步 |
| `tvm_runner` | Linux | 复用当前 trusted current artifact 执行链 |
| `safety_guard` | RTOS/Bare Metal | 作业授权、心跳检测、安全停止、故障上报 |
| `audit_logger` | Linux | 落盘 job manifest、fault code、执行结论 |

## 5. 接口模型

### 5.1 传输层选择

建议采用：

- Linux：`rpmsg_char` 用户态端点
- 从核：OpenAMP endpoint + 固定消息结构

第一版不建议使用文本 JSON 作为线上协议，建议采用：

- 固定头部
- 固定长度或小体积变长消息体
- 小于 `256B` 的控制消息

理由：

1. RTOS 侧实现简单
2. 容易做 CRC 与边界检查
3. 更贴近赛题中的工程实现

### 5.2 统一消息头

建议定义控制消息头：

```c
typedef struct {
    uint32_t magic;          // 固定 0x53434F4D: "SCOM"
    uint16_t version;        // 协议版本，当前 1
    uint16_t msg_type;       // 消息类型
    uint32_t seq;            // 单向递增序号
    uint32_t job_id;         // 当前作业编号
    uint32_t payload_len;    // 消息体长度
    uint32_t header_crc32;   // 头部 CRC32（除本字段外）
} sc_ctrl_hdr_t;
```

字段约束：

- `magic` 错误直接丢弃
- `version` 不兼容直接回 `DENY_UNSUPPORTED_VERSION`
- `seq` 必须非回退
- `job_id` 在同一执行窗口中唯一
- `payload_len` 超出上限直接拒收

### 5.3 消息类型定义

| `msg_type` | 名称 | 方向 | 用途 |
|---|---|---|---|
| `0x01` | `JOB_REQ` | Linux -> RTOS | 作业执行申请 |
| `0x02` | `JOB_ACK` | RTOS -> Linux | `ALLOW` / `DENY` |
| `0x03` | `HEARTBEAT` | Linux -> RTOS | 运行中保活 |
| `0x04` | `HEARTBEAT_ACK` | RTOS -> Linux | 可选确认 |
| `0x05` | `JOB_DONE` | Linux -> RTOS | 作业结束 |
| `0x06` | `FAULT_REPORT` | RTOS -> Linux | 故障上报 |
| `0x07` | `SAFE_STOP` | RTOS -> Linux | 强制停止 |
| `0x08` | `STATUS_REQ` | Linux -> RTOS | 状态查询 |
| `0x09` | `STATUS_RESP` | RTOS -> Linux | 状态返回 |
| `0x0A` | `RESET_REQ` | Linux -> RTOS | 人工清故障 |
| `0x0B` | `RESET_ACK` | RTOS -> Linux | 清故障确认 |

### 5.4 消息体定义

#### 5.4.1 `JOB_REQ`

```c
typedef struct {
    uint32_t expected_sha256_hi;   // 可选：若只传摘要，可拆段或传 hash_id
    uint32_t expected_sha256_lo;
    uint32_t input_shape_n;
    uint32_t input_shape_c;
    uint32_t input_shape_h;
    uint32_t input_shape_w;
    uint32_t input_dtype;          // 1=float32
    uint32_t snr_db_x100;          // 例如 10.00dB -> 1000
    uint32_t payload_crc32;
    uint32_t deadline_ms;
    uint32_t expected_outputs;     // 1 或 300
    uint32_t flags;                // payload / reconstruction / demo / benchmark
} sc_job_req_t;
```

说明：

- 第一版如不方便传完整 SHA，可传 `sha_manifest_id` 或 64-bit 缩略索引
- `flags` 用于区分 payload benchmark 与 real reconstruction
- `deadline_ms` 由 Linux 发出，从核只做超时监护

#### 5.4.2 `JOB_ACK`

```c
typedef struct {
    uint32_t decision;      // 0=DENY, 1=ALLOW
    uint32_t fault_code;    // 若拒绝则携带故障码
    uint32_t guard_state;   // 当前 guard 状态
} sc_job_ack_t;
```

#### 5.4.3 `HEARTBEAT`

```c
typedef struct {
    uint32_t runtime_state;     // PRECHECK/RUNNING/FINALIZING
    uint32_t elapsed_ms;
    uint32_t completed_outputs; // 当前已完成输出数
    uint32_t progress_x100;     // 0-10000
} sc_heartbeat_t;
```

#### 5.4.4 `JOB_DONE`

```c
typedef struct {
    uint32_t result_code;       // 0=success, nonzero=failed
    uint32_t output_count;
    uint32_t result_crc32;
    uint32_t reserved;
} sc_job_done_t;
```

#### 5.4.5 `FAULT_REPORT`

```c
typedef struct {
    uint32_t fault_code;
    uint32_t fault_arg0;
    uint32_t fault_arg1;
    uint32_t guard_state;
} sc_fault_report_t;
```

#### 5.4.6 `STATUS_RESP`

```c
typedef struct {
    uint32_t guard_state;
    uint32_t active_job_id;
    uint32_t last_fault_code;
    uint32_t heartbeat_ok;
    uint32_t sticky_fault;
    uint32_t total_fault_count;
} sc_status_resp_t;
```

## 6. 状态机设计

### 6.1 Linux `orchestrator` 状态机

```text
IDLE
  -> PRECHECK
  -> REQUESTING
  -> ALLOWED
  -> RUNNING
  -> FINALIZING
  -> DONE

异常分支：
REQUESTING -> DENIED
RUNNING -> SAFE_STOPPED
RUNNING -> FAILED
FINALIZING -> FAILED
```

状态含义：

| 状态 | 含义 |
|---|---|
| `IDLE` | 空闲，可接受新作业 |
| `PRECHECK` | 本地做 SHA、shape、dtype、manifest、参数检查 |
| `REQUESTING` | 向从核发送 `JOB_REQ` 等待授权 |
| `ALLOWED` | 已获授权，准备启动 TVM current 任务 |
| `RUNNING` | 正在执行，周期性发送心跳 |
| `FINALIZING` | 落盘、核验输出、生成 manifest |
| `DONE` | 成功结束 |
| `DENIED` | 从核拒绝，未进入 TVM 执行 |
| `SAFE_STOPPED` | 收到从核 `SAFE_STOP`，进入受控停止 |
| `FAILED` | 本地执行失败或输出不完整 |

### 6.2 RTOS `safety_guard` 状态机

```text
BOOT
  -> READY
  -> JOB_ACTIVE
  -> WAIT_DONE
  -> READY

异常分支：
READY -> DENY_PENDING
JOB_ACTIVE -> FAULT_LATCHED
WAIT_DONE -> FAULT_LATCHED
FAULT_LATCHED -> READY (仅在 RESET_REQ 后)
```

状态含义：

| 状态 | 含义 |
|---|---|
| `BOOT` | 上电初始化 |
| `READY` | 可接受新作业 |
| `JOB_ACTIVE` | 当前有活动作业，监控 heartbeat/deadline |
| `WAIT_DONE` | 已接收停止信号，等待 `JOB_DONE` |
| `DENY_PENDING` | 触发拒绝，待 Linux 读取状态 |
| `FAULT_LATCHED` | 锁存故障，需人工 `RESET_REQ` 清除 |

### 6.3 关键状态迁移条件

| 条件 | 迁移 |
|---|---|
| `JOB_REQ` 校验通过 | `READY -> JOB_ACTIVE` |
| `JOB_REQ` 校验失败 | `READY -> DENY_PENDING` |
| 心跳超时 | `JOB_ACTIVE -> FAULT_LATCHED` |
| deadline 超时 | `JOB_ACTIVE -> FAULT_LATCHED` |
| 非法帧/坏 CRC | `READY/JOB_ACTIVE -> FAULT_LATCHED` |
| `JOB_DONE` 成功且无故障 | `JOB_ACTIVE -> READY` |
| `RESET_REQ` | `FAULT_LATCHED -> READY` |

## 7. 故障码设计

建议统一故障码前缀为 `F0xx`：

| 故障码 | 名称 | 说明 | 默认动作 |
|---|---|---|---|
| `F001` | `ARTIFACT_SHA_MISMATCH` | artifact SHA 不匹配 trusted current | `DENY` |
| `F002` | `INPUT_CONTRACT_INVALID` | batch/shape/dtype 不合法 | `DENY` |
| `F003` | `HEARTBEAT_TIMEOUT` | 心跳超时 | `SAFE_STOP` + `FAULT_LATCHED` |
| `F004` | `CONTROL_CRC_ERROR` | 控制帧 CRC 错误 | `DENY` 或 `FAULT_LATCHED` |
| `F005` | `OUTPUT_INCOMPLETE` | 输出数量或 manifest 不完整 | `FAILED` + `FAULT_REPORT` |
| `F006` | `DEADLINE_EXCEEDED` | 超过 `deadline_ms` | `SAFE_STOP` |
| `F007` | `UNSUPPORTED_VERSION` | 协议版本不兼容 | `DENY` |
| `F008` | `DUPLICATE_JOB_ID` | 重复 job_id | `DENY` |
| `F009` | `ILLEGAL_PARAM_RANGE` | SNR、shape、计数越界 | `DENY` |
| `F010` | `MANUAL_SAFE_STOP` | 人工触发安全停止 | `SAFE_STOP` |

建议规则：

- `F001/F002/F007/F008/F009`：进入前置拒绝
- `F003/F006/F010`：运行中强制安全停止
- `F004/F005`：根据场景落 `DENY` 或 `FAULT_LATCHED`

## 8. 关键时序

### 8.1 正常路径

```text
Linux orchestrator                RTOS safety_guard
--------------------------------------------------------------
PRECHECK ok
JOB_REQ ----------------------->  validate policy
                                 ALLOW
<------------------------------ JOB_ACK(ALLOW)
start TVM current
HEARTBEAT -------------------->  refresh watchdog
HEARTBEAT -------------------->  refresh watchdog
...
JOB_DONE(success) ------------>  clear active job
STATUS_REQ ------------------->  final state query
<------------------------------ STATUS_RESP(READY)
```

### 8.2 心跳超时路径

```text
Linux orchestrator                RTOS safety_guard
--------------------------------------------------------------
JOB_REQ ----------------------->  ALLOW
<------------------------------ JOB_ACK(ALLOW)
start TVM current
HEARTBEAT -------------------->  watchdog ok
(heartbeat lost)
                                 timeout detected
<------------------------------ SAFE_STOP(F003)
stop current job
JOB_DONE(failed) ------------->  latch fault
```

### 8.3 错误 artifact 路径

```text
Linux orchestrator                RTOS safety_guard
--------------------------------------------------------------
local precheck fail OR
JOB_REQ(expected_sha mismatch)->  validate fail
<------------------------------ JOB_ACK(DENY, F001)
no TVM run
```

### 10.1 功能测试

| 用例 ID | 名称 | 前置条件 | 操作 | 预期结果 | 证据 |
|---|---|---|---|---|---|
| `TC-001` | payload 正常作业 | trusted SHA、合法参数、从核 READY | 提交 payload 作业 | `ALLOW`，作业完成，`JOB_DONE=success` | report + status log |

| `TC-004` | 输入契约违规拒绝 | 构造 `batch=4` 或非法 shape | 提交 `JOB_REQ` | 从核 `DENY(F002)` | guard log |
| `TC-005` | heartbeat 正常保活 | job 正常运行 | 每 100ms 发送 heartbeat | 从核无 fault，job 完成 | heartbeat trace |

| `TC-009` | 输出不完整失败 | 模拟中途写盘失败 | 执行 reconstruction | `JOB_DONE(failed)`，记录 `F005` | output count log |
| `TC-010` | sticky fault 恢复 | 先触发 `F003/F004` | 发送 `RESET_REQ` | `RESET_ACK`，状态回 READY | state transition log |
| `TC-011` | status 查询 | 从核任意状态 | 发送 `STATUS_REQ` | 返回状态、最后故障、fault count | status dump |
| `TC-012` | 连续作业稳定性 | 全链路可运行 | 连跑 10 次合法作业 | 无状态泄漏、无 job_id 冲突 | summary report |

### 10.2 FIT 故障注入测试

| FIT ID | 注入故障 | 注入方式 | 预期结果 | 对应风险 |
|---|---|---|---|---|
| `FIT-01` | 错误 artifact | 替换 `.so` 或改 expected SHA | 拒绝执行，`F001` | 未知 artifact 执行风险 |
| `FIT-02` | 输入契约破坏 | 提交 `batch=4`/非法 shape | 前置拦截，`F002` | 输入契约违规风险 |
| `FIT-03` | 主控失活 | 暂停 heartbeat | 从核 `SAFE_STOP`，fault latch | 主控失活风险 |
| `FIT-04` | 参数/帧篡改 | 发错 CRC/非法控制帧 | 丢弃任务并告警 | 非法数据/参数篡改风险 |
| `FIT-05` | 结果不完整 | 模拟写盘失败/删部分输出 | 标记失败，保留故障记录 | 结果不完整风险 |

### 10.3 通过标准

OpenAMP 最小闭环阶段建议采用以下退出条件：

1. `TC-001`、`TC-002`、`TC-003`、`TC-004`、`TC-006`、`TC-010` 全部通过
2. 至少 3 个 FIT 样例正式跑通并有证据落盘
3. 从核能稳定完成 `DENY`、`SAFE_STOP`、`RESET` 三种控制闭环
4. 文档能形成“风险项 -> 控制措施 -> FIT -> 证据”的闭环链条

## 12. 最小实现路径

### 阶段 A：接口先行

1. 固化本文档中的消息类型、状态机、故障码
2. 在 Linux 侧先做 mock transport 验证消息与状态机

# 飞腾赛题对齐与系统重构建议

- 对应赛事：第八届全国大学生集成电路创新创业大赛（飞腾杯）
- 文档日期：2026-03-13
- 文档性质：内部重构建议（基于 2026-03-13 赛题对齐建议整理）
- 适用范围：TVM 主线、OpenAMP 引入、Demo 改造、测试覆盖/FIT/风险分析

---

## 1. 结论摘要

当前阶段最重要的，不是推翻现有 TVM 路线，也不是继续单点追求更低的 payload 延迟，而是把已经成立的高性能解码成果升级为：

**基于飞腾平台的安全可靠图像语义通信系统。**

建议将项目主线重定义为：

- TVM：高性能解码主线
- OpenAMP：异构监护、心跳控制、故障隔离、安全停止
- Demo：把端到端全链路与故障注入演示跑出来
- 测试文档：补齐功能覆盖、风险分析、风险管控、FIT 故障注入与复杂环境验证

在当前赛题约束下，项目的主要矛盾已经从“能不能继续提速”转变为：

1. 能不能把现有 TVM 成果转化成符合飞腾赛题要求的中阶闭环
2. 能不能补齐高阶任务要求的安全可靠设计、覆盖分析、风险分析和故障注入验证
3. 能不能把结果以可信、系统化、可答辩的方式展示出来

## 2. 当前可信底座

对外口径建议统一到以下正式结果：

- trusted current SHA：`6f236b07...6dc1`
- 正式 payload 口径：`1846.9 -> 130.219 ms`
- 正式真实端到端重建口径：`1850.0 -> 230.339 ms/image`
- 两条链路均已执行 artifact SHA guard，且 `match=true`

这意味着当前工程已经具备一套非常硬的底座：

1. safe runtime 已打通
2. target 已收敛
3. warm-start incremental tuning 已跑通
4. benchmark 口径已分层
5. artifact SHA guard 已上线
6. trusted current 已经有正式成对验证结果

因此后续重构的原则应当是：

- 尽量不动现有 TVM current 可信执行链
- 在外围增加赛题要求的异构控制、风险治理和可验证闭环

---

## 3. 应保留的核心资产

### 3.1 TVM current 主线必须保留

当前 TVM 路线已经形成完整工程链：

- safe runtime
- target 收敛
- warm-start incremental
- payload benchmark
- 真实端到端 reconstruction benchmark
- artifact SHA guard

这是当前项目最强的技术壁垒，也是答辩中最有说服力的硬结果来源。

### 3.2 MNN 保留为动态尺寸备线

MNN 仍然有价值，但现阶段建议只作为：

- 动态尺寸支持备线
- 对比路线
- 量化替代路线（若 TVM 量化风险过高）

不建议让 MNN 抢主叙事，因为当前公开结果仍明显弱于 TVM 主线。

### 3.3 artifact SHA guard 上升为安全设计卖点

artifact SHA guard 原本属于实验治理手段，但现在应上升为正式的安全能力：

- trusted artifact 身份校验
- 非法镜像拒绝执行
- 部署版本可追溯
- 故障注入可验证

### 3.4 历史真实故障样本应转化为 FIT 案例

例如已经发生过的：

- `batch=4` 与模型固定 `batch=1` 不匹配

这类真实故障不要隐藏，应改写为：

- 输入契约校验
- 故障前置拦截
- FIT 故障注入样例

## 4. 作品主线重定义

建议项目主标题调整为类似以下方向：

**《基于飞腾平台的安全可靠图像语义通信系统：TVM 高性能解码、OpenAMP 异构监护与端到端验证》**

这样可以把项目与赛题要求自然对应起来：

- 基础任务：图像语义通信系统
- 中阶任务：OpenAMP 异构多核协同
- 高阶任务：安全可靠设计、风险分析、测试覆盖、FIT、复杂环境验证

重定义之后，项目就不再是“一个单纯的 TVM/MNN 优化项目”，而是“一个在飞腾平台上具备性能、异构协同和安全可靠能力的系统作品”。

## 5. 技术路线建议

### 5.1 总原则

**不要强行让 TVM 与 OpenAMP 深度绑定。**

更稳妥、更适合当前进度的做法是：

**数据面/控制面分离。**

### 5.2 数据面保持不动

现有可信数据面尽量保持不变：

- 上位机：Encoder、加噪、OpenSSH/SCP 传 latent
- 飞腾派 Linux 侧：继续运行 trusted current TVM artifact
- 继续沿用 safe runtime、`REMOTE_TVM_PYTHON`、`current_real_reconstruction.py`
- 继续沿用 `INFERENCE_CURRENT_EXPECTED_SHA256` 和 SHA guard

也就是说：

- 当前可用的 TVM current 执行链不应被 OpenAMP 重写
- OpenAMP 不负责承载大张量数据流

### 5.3 控制面交给 OpenAMP

建议 Linux 主核和从核的职责如下：

- Linux 主核：
  - `guard_client`
  - `orchestrator`
  - TVM current artifact 调度与执行
- RTOS/Bare Metal 从核：
  - `safety_guard`
  - 心跳监控
  - 参数校验
  - 安全停止
  - 故障计数与状态维持

OpenAMP/RPMsg 通道只传小消息，不传大张量，例如：

- `job_id`
- `expected_sha256`
- `input_shape`
- `snr_db`
- `payload_crc32`
- `deadline_ms`
- `heartbeat`
- `fault_code`
- `job_done`

这样做的优点：

1. 不破坏现有 TVM trusted current 路线
2. OpenAMP 的角色清晰落在异构协同与安全监护上
3. 高阶任务中的风险分析、覆盖分析、FIT 都能自然挂到控制面

## 6. 分阶段执行方案

### 6.1 第一阶段：冻结可信底座

目标：

- 统一 trusted current SHA
- 统一 benchmark 数字
- 统一 benchmark 口径
- 清理旧文档中的过时数值

需要统一替换的重点内容：

- `1946b08e...` -> `6f236b07...6dc1`
- `152.4 ms`、`152.36 ms` 等旧 current payload 数值
- `255.931 ms/image` 等上一代 current e2e 数值
- Demo 中使用的旧 artifact SHA 和预测值

要求：

- payload-only 与 real reconstruction 严禁混写
- 所有文档明确区分“正式 payload 口径”和“正式真实端到端口径”

### 6.2 第二阶段：实现 OpenAMP 最小可行闭环

建议只实现最小控制闭环，不做复杂跨核数据搬运：

1. Linux 解码前发送 `JOB_REQ`
2. RTOS 校验策略并返回 `ALLOW` 或 `DENY`
3. 解码过程中 Linux 周期性发送 `HEARTBEAT`
4. RTOS 心跳超时则回 `SAFE_STOP`
5. Linux 解码结束后发送 `JOB_DONE`
6. RTOS 记录事件计数、故障码和状态

该阶段完成后，OpenAMP 就从“文档引用”变成“系统中实际工作的中阶能力”。

### 6.3 第三阶段：补齐安全可靠设计与测试覆盖

建议定义 6 个关键功能：

1. 图像语义编码与传输启动
2. trusted current artifact 选择与 SHA 校验
3. TVM current 解码执行
4. OpenAMP 异构监护
5. 异常拒绝/安全停止
6. 重建结果落盘与日志追溯

然后构建：

- 功能规范表
- 测试用例表
- 覆盖矩阵

要求形成“功能项 -> 测试项 -> 结果项”的可追溯闭环。

### 6.4 第四阶段：补 5 个高价值风险项与 FIT

建议至少覆盖以下 5 类风险：

#### 风险 1：未知 artifact 执行风险

- 风险：远端 `.so` 漂移或被错误替换
- 管控：`INFERENCE_CURRENT_EXPECTED_SHA256` + SHA guard
- FIT：故意替换错误 `.so` 或修改 manifest，验证系统拒绝执行并报警

#### 风险 2：输入契约违规风险

- 风险：batch/shape/dtype 不匹配
- 管控：job 前置校验
- FIT：复现 `batch=4` 与固定 `batch=1` 不匹配案例，要求在进入 TVM 前被拦截

#### 风险 3：主控失活风险

- 风险：Linux 解码进程卡死或失联
- 管控：OpenAMP 心跳 + deadline
- FIT：人工暂停 Linux 心跳线程，验证 RTOS 进入 safe state，并拒绝后续任务

#### 风险 4：非法参数/控制帧篡改风险

- 风险：SNR、shape、job_id、payload 元数据被改写
- 管控：CRC32、序号、边界检查、会话 token
- FIT：发送非法控制帧或错误 CRC，验证系统告警并丢弃

#### 风险 5：结果不完整风险

- 风险：PNG 输出不完整、输出数量不一致、中途写盘失败
- 管控：结果 manifest、原子写盘、output_count 核验
- FIT：模拟写盘失败，验证系统标记失败并保留安全状态

### 6.5 第五阶段：重做 Demo 叙事

Demo 的目标不再是“秀所有功能”，而是突出：

- 可信启动
- 高性能重建
- 正式口径对比
- 故障注入与恢复

建议改为四幕式结构：

#### 第一幕：可信启动

界面显示：

- 飞腾派在线
- OpenAMP 从核在线
- trusted current SHA
- 当前 target
- 当前 runtime

#### 第二幕：一键重建

展示：

- 原图
- 重建图
- 编码/传输/解码/总计耗时
- PSNR/SSIM

#### 第三幕：正式基线对比

明确显示两条正式口径：

- 正式 payload：`1846.9 -> 130.219 ms`
- 正式真实端到端：`1850.0 -> 230.339 ms/image`

不要在主画面混用其他非正式口径。

#### 第四幕：故障注入与恢复

建议至少提供三个按钮：

- 错误 SHA 注入
- 心跳超时注入
- 非法参数注入

注入后要求展示：

- 从核状态灯变红
- fault code 输出
- 系统拒绝执行或进入 safe state
- 恢复后重新回到绿色可运行状态

## 7. 优先级重排

### 7.1 新的 P0

必须立刻做：

1. 冻结最新 trusted current 数字、SHA 与口径
2. 更新 Demo 与文档中的旧数值
3. 跑通 OpenAMP 最小控制面
4. 写出 5 个风险项与 5 组 FIT
5. 至少正式跑通其中 3 组 FIT

这一阶段的目标不是继续提速，而是：

**完成赛题对齐。**

### 7.2 新的 P1

决定项目是否具备冲奖说服力：

1. PSNR/SSIM/LPIPS
2. 算子级 profiling
3. CPU/内存资源画像
4. 多 SNR 曲线
5. big.LITTLE 吞吐实验

这一层对应的是评委最容易追问、原方案也明确指出缺失的内容。

### 7.3 新的 P2

作为一等奖差异化，但必须后置：

1. big.LITTLE 异构流水线正式数据
2. 跨框架对比
3. 更深的 TVM hotspot 深搜
4. 更激进的手写 TIR/硬核方向

不建议当前优先投入：

- 自定义 Pass
- 动态 TVM
- 模型剪枝重训

这些方向要么 ROI 低，要么 API 风险高，要么已接近独立子项目。

## 8. 关键注意事项

### 8.1 飞腾派核型表述需要统一

现有文稿中对飞腾派核型存在前后不一致的问题。

建议统一策略：

- 硬件介绍页：使用飞腾官方表述
- TVM 优化页：单独说明当前 LLVM/TVM codegen 稳定 target 收敛为 `cortex-a72 + neon`
- 演示页：避免直接把整块板简单等同为 “A72/A55 big.LITTLE”

### 8.2 不要把 OpenAMP 做成高风险重写工程

OpenAMP 的目标是：

- 控制
- 监护
- 生命周期
- 小消息协同

而不是：

- 重写 TVM 执行路径
- 让 RTOS 承担推理主执行
- 让 RPMsg 承载大规模张量传输

### 8.3 文档与 Demo 必须全部切到最新 trusted current

尤其要清理：

- 旧 SHA `1946...`
- 旧 current payload 数值
- 旧 reconstruction 数值
- 旧的预计值和非正式值

否则会直接损伤作品的可信性。

## 9. 最终主叙事建议

建议统一采用如下主叙事：

> 我们不是只把 cGAN 解码器在飞腾派上跑快了。我们实现的是一个基于飞腾平台的安全可靠图像语义通信系统：上位机负责语义编码与信道扰动，飞腾派 Linux 侧基于 TVM trusted current artifact 完成高性能解码，从核基于 OpenAMP 承担异构监护、心跳检测、参数校验和安全停止；系统通过 SHA guard 保证部署可信，通过覆盖分析保证功能可测，通过 FIT 故障注入验证复杂环境下的安全可靠运行。

这条叙事同时保住了现有最强的 TVM 成果，也把作品自然对齐到飞腾赛题真正关注的中阶与高阶要求。

# Demo 演示软件改进方案

- 文档日期：2026-03-15
- 关联文档：飞腾赛题对齐与系统重构建议_2026-03-13.md
- 目标：将现有"静态证据看板"升级为"可交互四幕答辩演示系统"

---

## 1. 现有 Demo 问题诊断

### 1.1 功能架构层面

| 问题 | 现状 | 影响 |
|---|---|---|
| **无可操作按钮** | 整个 demo 仅有"刷新证据"和"读取板卡状态"两个按钮，所有数据来源于预存的 markdown/JSON 报告文件 | 评委看到的是一个只读报告浏览器，无法感受"系统在工作" |
| **不能实时测量端到端加速比** | 性能数据从 `inference_compare_*.md` 静态解析，无法发起真实推理并计时 | 丢失最有冲击力的"一键跑出加速比"演示效果 |
| **无故障注入交互** | 项目计划要求的"错误 SHA 注入 / 心跳超时注入 / 非法参数注入"三个按钮完全缺失 | 高阶任务中最核心的 FIT 演示无法现场操作，只能翻文档 |
| **无图像重建对比** | 缺少原图 vs 重建图的可视化展示，无 PSNR/SSIM 显示 | 语义通信系统最直观的效果无法呈现 |
| **四幕结构未落地** | 项目计划定义了四幕（可信启动→一键重建→基线对比→故障注入），但 UI 是一个平铺长页面，没有幕的概念 | 答辩时无法按剧本推进，评委体验混乱 |

### 1.2 UI/UX 层面

| 问题 | 技术原因 | 影响 |
|---|---|---|
| **中文字体重叠** | CSS 字体栈为纯西文字体 `"Iowan Old Style", "Palatino Linotype", "Book Antiqua", serif`，无中文 fallback；标题使用 `line-height: 0.95` 对中文过于紧凑 | 中文标题和正文行间距不足，笔画密集处发生视觉叠压 |
| **元素堆叠混乱** | `.hero::after` 伪元素绝对定位造成遮盖；多层 `backdrop-filter: blur()` 叠加性能差；`.background-glow` 固定定位在窄屏下溢出 | 滚动时卡片被装饰层遮盖，视觉层次混乱 |
| **信息密度过高** | 单页平铺了板卡状态、启动命令、里程碑、FIT、性能、材料索引六大区域，无折叠无导航 | 评委一眼看不到重点，滚动量大 |
| **响应式不足** | `@media` 断点仅处理了栅格，未处理字号、间距、卡片内文本溢出 | 投影/大屏场景下字过小，笔记本屏幕下卡片内容溢出 |

## 2. 改进总体方案

### 2.1 核心设计理念

将 demo 从**"静态证据浏览器"**重构为**"四幕可交互答辩演示系统"**：

- 每一幕对应项目计划中定义的一个答辩阶段
- 每一幕都有明确的**操作按钮**和**实时反馈区域**
- 预存证据作为 fallback，而非唯一数据来源
- 大字号、高对比度、答辩投影友好

### 2.2 四幕结构定义

```
┌─────────────────────────────────────────────────────────────┐
│                    顶部导航 + 系统状态指示灯                    │
├───────────┬───────────┬───────────┬───────────────────────────┤
│  第一幕    │  第二幕    │  第三幕    │  第四幕                    │
│ 可信启动   │ 一键重建   │ 基线对比   │ 故障注入与恢复              │
└───────────┴───────────┴───────────┴───────────────────────────┘
```

| 幕 | 标题 | 核心交互 | 关键展示 |
|---|---|---|---|
| 第一幕 | 可信启动 | 按钮：探测板卡 / 检查 OpenAMP 从核 | 飞腾派在线状态灯、remoteproc 状态、trusted SHA、runtime/target |
| 第二幕 | 一键重建 | 按钮：选图 + 开始推理 | 原图 vs 重建图并排、编码/传输/解码/总耗时分段计时条、PSNR/SSIM |
| 第三幕 | 基线对比 | 按钮：运行 baseline / 运行 current / 全部运行 | 两条正式口径大字展示、加速比进度条动画、实时计时 |
| 第四幕 | 故障注入 | 三个按钮：错误SHA / 心跳超时 / 非法参数 + 恢复按钮 | 从核状态灯（绿→红→绿）、fault code、拒绝/safe state 日志流 |

## 3. 详细改进项

### 3.1 后端改进（server.py / 新增 API）

#### 3.1.1 新增 API 端点

```
POST /api/probe-board              # 已有，保留
POST /api/run-inference             # 新增：触发真实推理
POST /api/run-baseline              # 新增：运行 baseline 推理
POST /api/inject-fault              # 新增：故障注入
POST /api/recover                   # 新增：故障恢复
GET  /api/system-status             # 新增：实时系统状态（SSE 或轮询）
GET  /api/snapshot                  # 已有，保留
```

#### 3.1.2 `/api/run-inference` 设计

请求体：
```json
{
  "image_index": 0,
  "mode": "current"
}
```

响应体（流式或最终）：
```json
{
  "status": "success",
  "original_image_b64": "...",
  "reconstructed_image_b64": "...",
  "timings": {
    "encode_ms": 45.2,
    "transmit_ms": 12.8,
    "decode_ms": 130.5,
    "total_ms": 230.3
  },
  "quality": {
    "psnr_db": 28.7,
    "ssim": 0.912
  },
  "artifact_sha": "6f236b07..."
}
```

实现方案：
- 调用项目中已有的 `current_real_reconstruction.py` 脚本
- 通过 SSH 远程在飞腾派上执行，捕获 stdout 解析结果
- 本地编码阶段在主机侧完成
- 图像以 base64 返回前端展示
- 如果飞腾派不可达，fallback 到预存结果并在 UI 标注"[离线数据]"

#### 3.1.3 `/api/inject-fault` 设计

请求体：
```json
{
  "fault_type": "wrong_sha" | "heartbeat_timeout" | "illegal_param"
}
```

响应体：
```json
{
  "status": "injected",
  "fault_type": "wrong_sha",
  "board_response": {
    "decision": "DENY",
    "fault_code": "ARTIFACT_SHA_MISMATCH",
    "guard_state": "FAULT_LATCHED"
  },
  "status_lamp": "red",
  "log_entries": [
    "[14:32:01] 发送 JOB_REQ (expected_sha=DEADBEEF...)",
    "[14:32:01] 收到 JOB_ACK: DENY, fault=ARTIFACT_SHA_MISMATCH",
    "[14:32:01] wrapper 拒绝放行 runner"
  ]
}
```

实现方案：
- 复用项目中已有的 `openamp_control_wrapper.py` 和 `openamp_rpmsg_bridge.py`
- 通过 RPMsg bridge 向从核发送构造好的错误 JOB_REQ
- 如果板卡不可达，播放预录的 FIT 日志序列作为 fallback 动画

#### 3.1.4 `/api/recover` 设计

- 向从核发送 `SAFE_STOP` → 等待 `STATUS_RESP` 回到 `READY`
- 更新状态灯为绿色

#### 3.1.5 `/api/system-status` 设计

使用 Server-Sent Events (SSE) 或 3 秒轮询，返回：
```json
{
  "board_online": true,
  "remoteproc_state": "running",
  "rpmsg_device": "/dev/rpmsg0",
  "guard_state": "READY",
  "last_fault_code": "NONE",
  "trusted_sha": "6f236b07...",
  "target": "cortex-a72 + neon",
  "runtime": "tvm"
}
```

## 4. Fallback 机制设计

答辩现场的最大风险是板卡不可达。每个交互功能都必须有 fallback：

| 功能 | 在线模式 | 离线 Fallback |
|---|---|---|
| 第一幕 探测板卡 | SSH 实时探板 | 展示最近一次成功探板的缓存 JSON，标注"[离线证据]" |
| 第二幕 一键重建 | 远程执行推理 | 加载预存的原图+重建图+计时数据，标注"[预录结果]" |
| 第三幕 基线运行 | 远程执行 benchmark | 展示正式收口数据，标注"[正式收口数据]" |
| 第四幕 故障注入 | RPMsg bridge 真实注入 | 播放预录日志序列动画，标注"[回放模式]" |

UI 中添加全局模式指示器：

```
🟢 在线模式 — 所有操作直连飞腾派
🟡 降级模式 — 部分操作使用预录数据
🔴 离线模式 — 全部使用预录证据
```

## 5. 预录数据准备清单

为保证 fallback 质量，需要在正式答辩前录制以下数据：

| 数据项 | 来源 | 存储格式 | 存储位置 |
|---|---|---|---|
| 3-5 张测试图原图 | CIFAR-10 / 自选 | PNG | `demo/assets/test_images/` |
| 对应重建图 | 飞腾派推理输出 | PNG | `demo/assets/reconstructed/` |
| 推理计时数据 | `current_real_reconstruction.py` 输出 | JSON | `demo/assets/prerecorded_inference.json` |
| baseline 计时数据 | baseline 推理输出 | JSON | `demo/assets/prerecorded_baseline.json` |
| PSNR/SSIM 数据 | 质量评估脚本 | JSON | 同上合并 |
| FIT-01 日志序列 | FIT 执行日志 | JSON (时间戳+消息数组) | `demo/assets/prerecorded_fit01.json` |

## 10. 总结

现有 demo 的根本问题是**把答辩演示做成了证据浏览器**——虽然证据链完整，但评委无法通过交互感受到系统是"活的"。

改造的核心思路是：

1. **从浏览变为操作** — 每一幕都有按钮可以触发真实动作
2. **从文档变为可视化** — 图像对比、计时条、条形图、状态灯、日志流
3. **从平铺变为分幕** — 四幕结构与答辩节奏对齐
4. **从纯在线变为 fallback-safe** — 每个功能都有离线降级路径
5. **从西文排版变为中文友好** — 字体栈、行高、字间距全面修复

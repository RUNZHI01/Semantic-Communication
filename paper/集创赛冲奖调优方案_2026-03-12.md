## 2. 竞赛获奖评估

### 2.1 项目亮点（核心竞争力）

**（1）选题方向前沿且契合赛道。**
语义通信（Semantic Communication）是 6G 核心技术方向之一，在国内通信学术界正处于研究热点期。将其与飞腾派端侧部署结合，完整覆盖了飞腾杯"国产处理器 + AI 应用"的赛道主题。相比常见的"图像分类/目标检测部署"选题，语义通信的学术深度和应用前瞻性更突出。

**（2）TVM 优化深度超出多数本科参赛队水平。**
绝大多数参赛队对 TVM 的使用停留在"导入模型 → AutoTVM/Ansor → 编译"的标准流程。本项目的差异化体现在：

- **运行时级调试能力：** 通过 `gdb` 追踪定位 SIGILL 来源，重建 safe runtime 隔离依赖链——这属于系统级工程能力，不是简单调参能做到的。
- **target 收敛实验设计：** 4 种 target 配置的真机对比，并引入方差指标排除不稳定选项——体现了实验方法论。
- **warm-start incremental tuning：** 利用历史 DB 作为种子继续搜索——在 MetaSchedule 的实际工程使用中属于进阶用法。
- **artifact 治理机制：** SHA guard 防止远端产物漂移——表明团队意识到了实验可复现性的重要性。

**（3）双路线互补架构思路清晰。**
TVM 主攻静态形状深度优化（极致性能），MNN 主攻动态尺寸支持（灵活性），两条路线功能互补、不相互冲突。

**（4）实验方法论严谨。**
明确区分三类对比口径（current-only / payload-symmetric / 真实端到端），并以最保守的端到端指标作为最终结论，这种做法在参赛作品中非常罕见，可以有效抵御评审的"cherry-picking"质疑。

**（5）86% 的端到端提升是硬指标。**
300 张 PNG 的真实重建场景下 1830.3 ms → 255.931 ms，中位数改善 86.02%。这不是理论估算，而是真机可复验数据。

### 2.2 不足与差距分析

以下逐项分析当前技术方案在飞腾杯评审视角下的不足和潜在风险：

**（1）缺少量化技术（最大短板）。**

飞腾杯评审对端侧推理优化的考量维度中，量化是最高频的追问点之一。飞腾派的 Cortex-A72 完整支持 ARMv8 NEON 指令集，其中 INT8 点积指令（SQDMULH、SQRDMULH 等）和 FP16 half-precision 运算在硬件层面均可用。当前 TVM 路线和 MNN 路线全部运行在 FP32 精度下，意味着：

- NEON 128-bit 寄存器只能并行处理 4 个 FP32，但如果降到 INT8 可以并行 16 个——理论吞吐提升 4x。
- 模型权重的显存/内存占用未被压缩，对飞腾派有限的 4GB RAM 和 cache 造成不必要的压力。
- 评审很可能直接质疑："为什么不量化？量化后精度是否可接受？"

**当前没有量化，意味着丢掉了一个预期 2-4x 的加速空间，且文档无法回应评审关于量化的追问。**

**（2）MNN 优化深度不够。**

1.85x 的加速比在 MNN 的能力范围内偏低。MNN 本身提供了丰富的优化手段（FP16/INT8 量化、半自动搜索、Winograd 卷积、手写汇编 kernel），但从文档和代码来看，当前仅使用了模型转换和双 Interpreter 并行，未触及 MNN 的核心优化能力。如果评审追问 MNN 路线的具体优化措施，"模型转换 + 双 Interpreter"的回答可能显得单薄。

**（3）缺少图像重建质量验证。**

对语义通信系统来说，编译优化是否引入了精度损失是一个关键问题。TVM 编译理论上是语义等价变换（不改变计算精度），但：

- FP32 下不同的算子实现、不同的计算顺序可能导致浮点累积误差
- 如果后续做 INT8/FP16 量化，精度损失更需要显式验证
- 评审可能质疑："速度快了 86%，图像质量掉了多少？"

**当前文档没有任何 PSNR / SSIM / LPIPS 指标来证明优化后的图像重建质量。** 这是一个在评审眼中非常明显的遗漏。

**（4）缺少算子级热点分析。**

文档展示了整体的端到端时间提升，但没有展示"优化前后各算子分别耗时多少"的细粒度数据。评审可能追问：

- 瓶颈算子是哪些？卷积、反卷积、归一化各占多少比例？
- MetaSchedule 的搜索主要在哪些 task 上产生了收益？
- 如果继续优化，预期还有多少空间？

没有逐算子 profiling 数据，这些问题无法给出定量回答。

**（5）缺少跨框架横向对比。**

飞腾杯评审喜欢看到"为什么选 TVM/MNN 而不是其他方案"的论证。当前文档缺少与 NCNN（腾讯，ARM NEON 优化很强）、ONNX Runtime（微软，支持 ARM EP）、TFLite 等主流端侧推理框架的性能对比。没有对比数据，评审可能认为 TVM 的 86% 提升只是相对于未优化 baseline 的结果，而不能证明 TVM 在飞腾派上的方案优越性。

**（6）缺少资源分析（功耗、内存、CPU 占用）。**

飞腾派作为资源受限设备，评审关注的不仅是推理速度，还包括：

- 推理时的峰值内存占用（4GB RAM 中占了多少）
- CPU 利用率（4 核是否被有效利用）
- 功耗（持续推理时的板级功耗）
- `.so` 产物的文件大小

**（7）调优预算偏保守。**

500 trials 对于一个包含多层 conv2d + batch_norm + residual block 的 Generator 网络来说，搜索空间远未被充分探索。MetaSchedule 的搜索效率虽然高于 AutoTVM，但 500 trials 可能只覆盖了少数高权重 task，而中低权重 task 的 schedule 可能仍停留在默认配置。

## 4. P0 级任务：必做（直接影响获奖）

### 4.1 补充图像重建质量指标（PSNR / SSIM / LPIPS）

#### 4.1.1 目标

证明 TVM/MNN 编译优化后的图像重建质量与 PyTorch 原始推理结果一致（或精度损失在可接受范围内）。

#### 4.1.2 方法

利用已有的真实端到端重建实验输出（baseline 和 current 各 300 张 PNG），与 PyTorch 原始推理结果进行像素级对比。

**需要三组图像数据：**

1. **PyTorch 原始推理输出**（ground truth reference）：在上位机用原始 PyTorch 模型对同一批 latent 输入做推理
2. **TVM baseline 输出**：已有，位于飞腾派 `infer_outputs/..._baseline/reconstructions/`
3. **TVM current 输出**：已有，位于飞腾派 `infer_outputs/..._current/reconstructions/`

**质量指标计算脚本模板：**

```python
#!/usr/bin/env python3
"""compute_image_quality_metrics.py
对比 TVM 优化前后的图像重建质量。
"""
import argparse
import glob
import os

import numpy as np
from PIL import Image
from skimage.metrics import peak_signal_noise_ratio as psnr
from skimage.metrics import structural_similarity as ssim

def load_image_as_array(path):
    """加载 PNG 图像并转为 float64 numpy 数组（0-1 范围）"""
    img = Image.open(path).convert("RGB")
    return np.asarray(img, dtype=np.float64) / 255.0

def compute_metrics(ref_dir, test_dir, max_images=300):
    """逐张对比 ref_dir 和 test_dir 中的同名图像"""
    ref_files = sorted(glob.glob(os.path.join(ref_dir, "*.png")))
    results = {"psnr": [], "ssim": []}

    for ref_path in ref_files[:max_images]:
        fname = os.path.basename(ref_path)
        test_path = os.path.join(test_dir, fname)
        if not os.path.isfile(test_path):
            continue

        ref_img = load_image_as_array(ref_path)
        test_img = load_image_as_array(test_path)

        if ref_img.shape != test_img.shape:
            # 如果尺寸不一致（249x249 vs 256x256），裁剪到公共区域
            min_h = min(ref_img.shape[0], test_img.shape[0])
            min_w = min(ref_img.shape[1], test_img.shape[1])
            ref_img = ref_img[:min_h, :min_w, :]
            test_img = test_img[:min_h, :min_w, :]

        results["psnr"].append(psnr(ref_img, test_img, data_range=1.0))
        results["ssim"].append(ssim(ref_img, test_img, data_range=1.0,
                                     channel_axis=2))

    return results

def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--ref-dir", required=True, help="PyTorch 原始推理输出目录")
    parser.add_argument("--test-dir", required=True, help="TVM 优化后推理输出目录")
    parser.add_argument("--max-images", type=int, default=300)
    args = parser.parse_args()

    results = compute_metrics(args.ref_dir, args.test_dir, args.max_images)

    print(f"Images compared: {len(results['psnr'])}")
    print(f"PSNR  mean: {np.mean(results['psnr']):.2f} dB  "
          f"std: {np.std(results['psnr']):.2f} dB  "
          f"min: {np.min(results['psnr']):.2f} dB")
    print(f"SSIM  mean: {np.mean(results['ssim']):.4f}  "
          f"std: {np.std(results['ssim']):.4f}  "
          f"min: {np.min(results['ssim']):.4f}")

if __name__ == "__main__":
    main()
```

### 4.2 TVM MetaSchedule 增大调优预算

#### 4.2.1 目标

将 MetaSchedule 搜索预算从 500 trials 提升到 2000-5000 trials，预期在 payload 级推理上再压缩 10-30%。

### 4.3 TVM INT8 / FP16 量化

#### 4.3.1 目标

在飞腾派 Cortex-A72 的 NEON 指令集支持下，通过降低计算精度获得额外的推理加速，同时保持图像重建质量在可接受范围内。

#### 4.3.2 飞腾派硬件量化能力分析

飞腾派搭载的 Phytium D2000 处理器包含 Cortex-A72 核心，其 NEON 单元支持：

| 精度 | NEON 并行度 (128-bit) | 理论吞吐相对 FP32 |
|---|---|---|
| FP32 | 4 ops/cycle | 1x（基线） |
| FP16 | 8 ops/cycle | 2x |
| INT8 | 16 ops/cycle | 4x |

> 注：Cortex-A72 不具备 ARMv8.2 的原生 FP16 运算扩展（FPHP），FP16 加速主要来自内存带宽节省和权重压缩。INT8 则通过 NEON SMULL/SQDMULH 系列指令实现真正的低精度计算。

#### 4.3.3 量化方案

**方案 A：TVM Relay 级 Post-Training Quantization（PTQ）**

```python
import tvm
from tvm import relay

# 加载 ONNX 模型到 Relay IR
onnx_model = onnx.load("temp_simp.onnx")
mod, params = relay.frontend.from_onnx(
    onnx_model, shape={"input": [1, 32, 32, 32]}
)

# 准备校准数据集（50-100 张 latent 输入）
def calibrate_dataset():
    for latent_file in calibration_files:
        data = np.load(latent_file).astype("float32")
        yield {"input": data}

# INT8 PTQ 量化
with relay.quantize.qconfig(
    calibrate_mode="kl_divergence",     # KL 散度校准
    weight_scale="max",                  # 权重使用最大值校准
    skip_conv_layers=[0],                # 第一层保留 FP32
    global_scale=8.0,
    round_for_shift=True,
):
    qmod = relay.quantize.quantize(mod, params, dataset=calibrate_dataset())

# 编译量化后的模型
target = tvm.target.Target(
    '{"kind":"llvm","mtriple":"aarch64-linux-gnu",'
    '"mcpu":"cortex-a72","mattr":["+neon"],"num-cores":4}'
)
with tvm.transform.PassContext(opt_level=3):
    lib = relay.build(qmod, target=target, params=params)
lib.export_library("optimized_model_int8.so")
```

**方案 B：TVM Relax 级 FP16 降精度**

如果 INT8 量化导致图像质量不可接受，FP16 是更保守的选择：

```python
# 在 Relax pipeline 中加入 FP16 cast
from tvm.relax.transform import LegalizeOps, AnnotateTIROpPattern
from tvm.relax.transform import FuseOps, FuseTIR

# 将模型权重和计算降到 FP16
# 注：需确认 TVM 0.24dev 对 Relax FP16 的支持程度
```

**方案 C：混合精度（推荐）**

对精度敏感的层（第一层卷积、最后一层反卷积）保持 FP32，中间层使用 INT8/FP16：

```python
with relay.quantize.qconfig(
    calibrate_mode="kl_divergence",
    weight_scale="power2",
    skip_conv_layers=[0],          # 第一层 FP32
    skip_dense_layer=False,
    # 可指定 per-layer 配置
):
    qmod = relay.quantize.quantize(mod, params, dataset=calibrate_dataset())
```

#### 4.3.4 量化精度验证流程

1. 生成量化模型产物 `optimized_model_int8.so`
2. 在飞腾派上用同一批 300 张 latent 输入运行重建
3. 计算量化模型 vs PyTorch FP32 的 PSNR / SSIM / LPIPS
4. 设定精度阈值：
   - PSNR ≥ 28 dB：可接受（语义通信场景对绝对保真度要求不如超分辨率）
   - SSIM ≥ 0.90：可接受
   - 如果低于阈值，尝试提高 `skip_conv_layers` 或切换到 FP16 方案

#### 4.3.5 预期收益

| 量化方案 | 预期推理时间 | 预期加速（vs FP32 current） | 精度风险 |
|---|---|---|---|
| INT8 PTQ | 40-80 ms | 2-4x | 中等，需校准 |
| FP16 | 80-120 ms | 1.3-2x | 低 |
| 混合精度 INT8+FP32 | 60-100 ms | 1.5-2.5x | 低-中 |

### 6.3 TVM 动态形状探索（Shape Bucketing）

#### 6.3.1 目标

探索 TVM 对动态输入形状的支持方案，减少对 MNN 的依赖。

#### 6.3.2 方案：Shape Bucketing

为常见图像分辨率预编译多个静态模型：

```python
# 为常见分辨率各编译一份 .so
shape_buckets = [
    (1, 32, 16, 16),   # 128x128 图像对应的 latent
    (1, 32, 32, 32),   # 256x256 图像对应的 latent
    (1, 32, 64, 64),   # 512x512 图像对应的 latent
]

for shape in shape_buckets:
    # 为每个 shape 做独立的 ONNX 导出 + MetaSchedule 调优
    compile_for_shape(shape)
```

运行时根据输入大小动态选择最匹配的 bucket：

```python
def select_bucket(input_shape):
    """选择最小的能容纳输入的 bucket"""
    for bucket in sorted(shape_buckets):
        if all(i <= b for i, b in zip(input_shape, bucket)):
            return bucket
    return shape_buckets[-1]  # fallback 到最大 bucket
```

## 7. 硬核差异化方向（冲击一等奖）

> **定位说明：** P0-P2 是"补齐短板、达到获奖门槛"，本章的硬核方向是"制造差异化、拉开与其他队伍的差距"。飞腾杯评审大多来自芯片和体系结构领域，他们最认可的是"真正理解了硬件微架构并据此做优化"，而不是"调了一个框架的参数"。以下每个方向都旨在展示团队在编译器-硬件边界上的深度工程能力。

### 7.1 手写 TensorIR Schedule + ARM NEON 内联

#### 7.1.1 核心思路

MetaSchedule 自动搜索到的 schedule 不一定是全局最优的。对 Generator 网络中 top-1 / top-2 热点算子（conv2d、deconv），手写 TIR schedule 并嵌入 ARM NEON 向量化策略，目标是直接打败 MetaSchedule 自动搜索的最优结果。

#### 7.1.2 为什么这个方向硬核

- 绝大多数参赛队对 TVM 的使用停留在 `tune_relax()` 的 API 层面，不会触及 TensorIR schedule 原语。
- 手写 TIR schedule 需要同时理解三件事：（1）TVM 的 schedule 变换语义；（2）Cortex-A72 的 cache 层次（L1 32KB, L2 512KB-1MB）；（3）NEON 128-bit 寄存器的向量化约束。
- 在答辩中一句话就能镇住评审："我们不仅用了自动搜索，还对瓶颈算子手写了面向 Cortex-A72 cache 层次的 NEON 向量化 schedule，比自动搜索结果再快 20%。"

#### 7.1.3 技术方案

**步骤一：确定热点算子。**

通过 4.2 节的 MetaSchedule task summary 或 5.1 节的 profiling 数据，确定耗时占比最大的 1-2 个 PrimFunc。对于 LGJSCC-c3 的 Generator，预期热点为：

- `fused_nn_conv2d_add_nn_relu`（卷积 + BN + ReLU 融合算子）
- `fused_nn_conv2d_transpose_add`（反卷积算子）

**步骤二：为热点算子编写手工 TIR schedule。**

以 conv2d NCHW 为例，针对 Cortex-A72 的硬件参数定制 schedule：

```python
import tvm
from tvm import tir
from tvm.script import tir as T

@T.prim_func
def conv2d_nchw_cortex_a72(
    X: T.Buffer((1, 32, 32, 32), "float32"),
    W: T.Buffer((64, 32, 3, 3), "float32"),
    Y: T.Buffer((1, 64, 30, 30), "float32"),
):
    """面向 Cortex-A72 + NEON 的 conv2d 手写模板"""
    for n, oc_outer, oh, ow in T.grid(1, 16, 30, 30):
        for oc_inner in T.vectorized(4):  # NEON 128-bit = 4 × FP32
            Y[n, oc_outer * 4 + oc_inner, oh, ow] = T.float32(0)
            for ic, kh, kw in T.grid(32, 3, 3):
                Y[n, oc_outer * 4 + oc_inner, oh, ow] += (
                    X[n, ic, oh + kh, ow + kw]
                    * W[oc_outer * 4 + oc_inner, ic, kh, kw]
                )
```

**步骤三：使用 `tvm.tir.Schedule` API 做精细变换。**

```python
sch = tvm.tir.Schedule(conv2d_nchw_cortex_a72)
block = sch.get_block("Y")

# --- Tiling：匹配 Cortex-A72 L1/L2 cache ---
# L1 数据缓存 32KB → 每次处理的数据块需 ≤ 32KB
# L2 缓存 512KB-1MB → 权重 tile 需 ≤ 512KB
n, oc, oh, ow, ic, kh, kw = sch.get_loops(block)

# 输出通道：外层 16 组，内层 4（NEON float32x4 向量宽度）
oc_o, oc_i = sch.split(oc, factors=[None, 4])

# 输出高度：外层分块，内层 6 行
# 6 行 × 30 列 × 4 字节 = 720B，远小于 L1 cache line
oh_o, oh_i = sch.split(oh, factors=[None, 6])

# 输出宽度：外层分块，内层 10 列
ow_o, ow_i = sch.split(ow, factors=[None, 10])

# 输入通道：外层分块，内层 8（每次从 L1 读 8 个通道的数据）
ic_o, ic_i = sch.split(ic, factors=[None, 8])

# --- Reorder：外层循环控制数据块切换，内层循环密集计算 ---
sch.reorder(n, oc_o, oh_o, ow_o, ic_o, oh_i, ow_i, ic_i, oc_i, kh, kw)

# --- 并行化：外层 oh_o 在多核间并行 ---
sch.parallel(oh_o)

# --- 向量化：最内层 oc_i 使用 NEON float32x4 ---
sch.vectorize(oc_i)

# --- 循环展开：3×3 kernel 的 kh/kw 全展开 ---
sch.unroll(kh)
sch.unroll(kw)

# --- 可选：输入通道内层也展开，利用寄存器 ---
sch.unroll(ic_i)
```

#### 7.1.4 Cortex-A72 微架构关键参数速查

| 参数 | 值 | 对 schedule 的影响 |
|---|---|---|
| L1 数据缓存 | 32 KB, 2-way | 内层 tile 数据量需 ≤ 32KB |
| L1 指令缓存 | 48 KB, 3-way | 循环展开后指令体积需 ≤ 48KB |
| L2 统一缓存 | 512KB - 1MB (共享) | 权重 tile + 输入 tile 需 ≤ L2 容量的 70% |
| NEON 寄存器 | 32 × 128-bit (Q0-Q31) | 最多同时保持 32 个 float32x4 向量 |
| NEON FP32 吞吐 | 2 FMLA/cycle (每核) | 理论峰值：8 FLOP/cycle/core |
| 发射宽度 | 3-wide decode, 8-deep OoO | 循环体内指令需 ≥ 3 条以填满流水线 |
| Cache line | 64 Bytes | tile 维度应为 64B 的倍数（16 个 FP32） |

|---|---|---|

| 文档价值 | **极高** | 评审追问"你们优化到了什么层次"时，直接展示 TIR 代码 |

### 7.2 big.LITTLE 异构流水线调度

#### 7.2.1 核心思路

飞腾派的 CPU 是 2 个 Cortex-A72 大核（1.8GHz）+ 2 个 Cortex-A55 小核（1.5GHz），属于 ARM big.LITTLE 异构多核架构。当前的 TVM 和 MNN 推理都将所有核心视为同构处理，没有区分大核和小核的计算特性差异。

硬核做法是设计**算子级异构流水线**：将计算密集型任务（conv2d 推理）绑定到大核，将 I/O 密集型任务（latent 读取、PNG 编码写盘、数据预处理）绑定到小核，形成计算与 I/O 的并行流水线。

#### 7.2.2 为什么这个方向硬核

- 这是 ARM big.LITTLE 架构的核心设计意图——大核高性能计算、小核低功耗辅助——但绝大多数端侧部署方案完全忽略了这一点。
- 飞腾杯评审看到"你针对飞腾派的 big.LITTLE 架构做了异构调度"，会认为你真正理解这块板子，而不是把它当成一个通用 ARM 盒子。
- 这个方向的实现难度相对较低（纯 Python/系统编程），但展示效果很好。

#### 7.2.3 流水线设计

```
时间步      大核 (Cortex-A72 × 2)               小核 (Cortex-A55 × 2)
───────────────────────────────────────────────────────────────────
  1     [TVM 推理: latent_0 → output_0]    [预加载: 读取 latent_1.pt]
  2     [TVM 推理: latent_1 → output_1]    [后处理: output_0 → PNG_0 写盘]
  3     [TVM 推理: latent_2 → output_2]    [后处理: output_1 → PNG_1 写盘]
  4     [TVM 推理: latent_3 → output_3]    [后处理: output_2 → PNG_2 写盘]
  ...         ...                                ...
```

不使用流水线时，300 张图片的处理是串行的：

```
读取 latent → TVM 推理 → PNG 编码写盘 → 读取下一个 latent → ...
```

使用异构流水线后，I/O 和推理并行，理论吞吐提升 30-50%。

#### 7.2.4 实现代码

```python
#!/usr/bin/env python3
"""big_little_pipeline.py
飞腾派 big.LITTLE 异构流水线推理引擎。
大核 (A72): TVM 模型推理
小核 (A55): I/O（latent 预加载 + PNG 后处理写盘）
"""

import ctypes
import multiprocessing as mp
import os
import queue
import time
from pathlib import Path

import numpy as np

# ── CPU 核心亲和性绑定 ──

def set_cpu_affinity(core_ids: list[int]):
    """将当前进程绑定到指定 CPU 核心（Linux sched_setaffinity）"""
    libc = ctypes.CDLL("libc.so.6", use_errno=True)
    mask = 0
    for cid in core_ids:
        mask |= (1 << cid)
    buf = ctypes.c_ulong(mask)
    ret = libc.sched_setaffinity(0, ctypes.sizeof(buf), ctypes.byref(buf))
    if ret != 0:
        raise OSError(f"sched_setaffinity failed for cores {core_ids}")

# 飞腾派核心映射（需根据实际 /sys/devices/system/cpu 确认）
BIG_CORES = [0, 1]    # Cortex-A72 @ 1.8GHz
LITTLE_CORES = [2, 3]  # Cortex-A55 @ 1.5GHz

# ── 大核工作进程：TVM 推理 ──

def big_core_inference_worker(
    artifact_path: str,
    input_queue: mp.Queue,
    output_queue: mp.Queue,
):
    """大核工作进程：绑定 A72，执行 TVM VM 推理"""
    set_cpu_affinity(BIG_CORES)

    import tvm
    from tvm import relax

    dev = tvm.cpu(0)
    lib = tvm.runtime.load_module(artifact_path)
    vm = relax.VirtualMachine(lib, dev)

    while True:
        item = input_queue.get()
        if item is None:  # 毒丸信号
            output_queue.put(None)
            break

        idx, latent_np = item
        tvm_input = tvm.nd.array(latent_np, dev)

        t0 = time.perf_counter()
        output = vm["main"](tvm_input)
        elapsed_ms = (time.perf_counter() - t0) * 1000

        output_np = output.numpy()
        output_queue.put((idx, output_np, elapsed_ms))

# ── 小核工作进程：I/O 预加载 + 后处理 ──

def little_core_preloader(
    latent_dir: str,
    input_queue: mp.Queue,
    total_files: int,
):
    """小核工作进程 A：预加载 latent .pt/.npy 文件到内存"""
    set_cpu_affinity(LITTLE_CORES)
    import torch

    latent_files = sorted(Path(latent_dir).glob("*.pt"))[:total_files]
    for idx, fpath in enumerate(latent_files):
        payload = torch.load(fpath, map_location="cpu", weights_only=True)
        if isinstance(payload, dict):
            latent = payload.get("quantized_latent", payload.get("latent"))
        else:
            latent = payload
        latent_np = np.asarray(latent, dtype=np.float32)
        if latent_np.ndim == 3:
            latent_np = np.expand_dims(latent_np, 0)
        input_queue.put((idx, latent_np))

    input_queue.put(None)  # 毒丸


def little_core_postprocessor(
    output_queue: mp.Queue,
    output_dir: str,
):
    """小核工作进程 B：将推理结果编码为 PNG 并写盘"""
    set_cpu_affinity(LITTLE_CORES)
    from PIL import Image

    os.makedirs(output_dir, exist_ok=True)
    times = []

    while True:
        item = output_queue.get()
        if item is None:
            break

        idx, output_np, infer_ms = item
        times.append(infer_ms)

        # output_np: [1, 3, H, W] float32 → [H, W, 3] uint8
        img_np = output_np[0].transpose(1, 2, 0)  # CHW → HWC
        img_np = np.clip(img_np * 255, 0, 255).astype(np.uint8)
        Image.fromarray(img_np).save(
            os.path.join(output_dir, f"recon_{idx:04d}.png")
        )

    return times

# ── 主流水线编排 ──

def run_pipeline(artifact_path, latent_dir, output_dir, num_images=300):
    """启动 big.LITTLE 异构流水线"""
    input_q = mp.Queue(maxsize=4)   # 预加载缓冲
    output_q = mp.Queue(maxsize=4)  # 推理结果缓冲

    # 启动小核预加载进程
    preloader = mp.Process(
        target=little_core_preloader,
        args=(latent_dir, input_q, num_images),
    )

    # 启动大核推理进程
    inferencer = mp.Process(
        target=big_core_inference_worker,
        args=(artifact_path, input_q, output_q),
    )

    preloader.start()
    inferencer.start()

    # 主进程（或另一个小核进程）做后处理
    set_cpu_affinity(LITTLE_CORES)
    from PIL import Image

    os.makedirs(output_dir, exist_ok=True)
    infer_times = []

    while True:
        item = output_q.get()
        if item is None:
            break
        idx, output_np, infer_ms = item
        infer_times.append(infer_ms)

        img_np = output_np[0].transpose(1, 2, 0)
        img_np = np.clip(img_np * 255, 0, 255).astype(np.uint8)
        Image.fromarray(img_np).save(
            os.path.join(output_dir, f"recon_{idx:04d}.png")
        )

    preloader.join()
    inferencer.join()

    import statistics
    print(f"Pipeline completed: {len(infer_times)} images")
    print(f"Inference median: {statistics.median(infer_times):.2f} ms")
    print(f"Inference mean:   {statistics.mean(infer_times):.2f} ms")

    return infer_times

if __name__ == "__main__":
    import argparse
    p = argparse.ArgumentParser()
    p.add_argument("--artifact", required=True)
    p.add_argument("--input-dir", required=True)
    p.add_argument("--output-dir", required=True)
    p.add_argument("--num-images", type=int, default=300)
    args = p.parse_args()
    run_pipeline(args.artifact, args.input_dir, args.output_dir, args.num_images)
```

| 指标 | 串行方式（当前） | big.LITTLE 流水线 | 改善 |
|---|---|---|---|
| 300 张图片总耗时 | ~77s (255ms × 300) | ~53-58s | **25-30% 吞吐提升** |
| 单张推理延迟 | 不变 | 不变 | 推理本身不变 |
| I/O 等待开销 | 与推理串行 | 被流水线隐藏 | **I/O 零等待** |
| CPU 大核利用率 | ~50%（等待 I/O 时空闲） | ~90%+ | **大核利用率翻倍** |

> 注：单张推理延迟不变，但批量处理的吞吐量显著提升。对于 300 张图片的端到端重建场景，这是最直接的系统级收益。

### 7.3 自定义 TVM 编译器 Pass（AWGN Fusion）

#### 7.3.1 核心思路

为语义通信模型编写一个**领域专用的 Relax 编译 Pass**，将标准 TVM pipeline 不会做的优化自动化。

在当前系统中，AWGN 噪声注入（信道模拟）在 Python 层完成：先在 CPU 上生成噪声、与 latent 相加，再把加噪后的结果拷贝给 TVM VM。这引入了一次额外的内存分配、numpy 计算和数据拷贝。

硬核做法：编写一个 Relax Pass，将 AWGN 噪声生成和解码器第一层 conv2d 融合为一个 kernel，实现"噪声注入-卷积"零拷贝流水。

#### 7.3.2 为什么这个方向硬核

- 在飞腾杯中，写过 TVM 编译器 Pass 的队伍极少（几乎为零）。
- 这直接证明团队对 TVM 编译栈有架构级理解，而不仅仅是 API 层面的使用。
- 在技术文档中可以单独开一节"领域专用编译优化"，这是强有力的创新点。

#### 7.3.3 技术方案

**方案 A：Relax IR 级 Pass（推荐）**

from tvm import relax
from tvm.ir.module import IRModule
from tvm.relax import Var, ShapeExpr, Call
from tvm.relax.transform import FunctionPass

**方案 B：TIR 级 Kernel Fusion（更底层，更硬核）**

直接在 TIR 层面编写一个融合了 AWGN + conv2d 的 PrimFunc：

```python
@T.prim_func
def fused_awgn_conv2d(
    latent: T.Buffer((1, 32, 32, 32), "float32"),
    weight: T.Buffer((64, 32, 3, 3), "float32"),
    sigma: T.float32,
    seed: T.int64,
    output: T.Buffer((1, 64, 30, 30), "float32"),
):
    """AWGN 噪声注入 + conv2d 融合 kernel。
    噪声在寄存器级别生成并直接参与卷积计算，无需额外内存分配。
    """
    noisy_latent = T.alloc_buffer((1, 32, 32, 32), "float32", scope="local")

    for n, c, h, w in T.grid(1, 32, 32, 32):
        # 在 local scope 中生成噪声并加到 latent 上
        noise = T.call_extern("float32", "tvm_awgn_sample", sigma, seed + n * 32768 + c * 1024 + h * 32 + w)
        noisy_latent[n, c, h, w] = latent[n, c, h, w] + noise

    for n, oc, oh, ow in T.grid(1, 64, 30, 30):
        output[n, oc, oh, ow] = T.float32(0)
        for ic, kh, kw in T.grid(32, 3, 3):
            output[n, oc, oh, ow] += (
                noisy_latent[n, ic, oh + kh, ow + kw]
                * weight[oc, ic, kh, kw]
            )
```

| 内存分配 | 额外分配 noisy_latent | 零分配（local scope） | **减少一次内存分配** |
| 数据拷贝 | numpy → TVM NDArray | 零拷贝 | **减少一次 host→device 拷贝** |

```python
import numpy as np

def build_real_channel_model(wifi_log_path):
    """从 WiFi 实测日志中提取信道参数"""
    rssi_values = parse_rssi_from_log(wifi_log_path)

    # 将 RSSI 转换为等效 SNR
    noise_floor_dbm = -95  # 典型 WiFi 噪声底
    snr_db = np.array(rssi_values) - noise_floor_dbm

    # 拟合信道参数分布
    snr_mean = np.mean(snr_db)
    snr_std = np.std(snr_db)
    fade_distribution = "rician"  # WiFi 室内信道通常为 Rician 衰落

    return {
        "snr_mean_db": snr_mean,
        "snr_std_db": snr_std,
        "fade_model": fade_distribution,
        "samples": len(rssi_values),
    }

def apply_real_channel(latent, channel_params):
    """用实测信道参数替代理想 AWGN 模型"""
    snr_db = np.random.normal(
        channel_params["snr_mean_db"],
        channel_params["snr_std_db"]
    )
    signal_power = np.mean(latent ** 2)
    noise_power = signal_power / (10 ** (snr_db / 10))

    # Rician 衰落（K-factor 取决于实测环境）
    K = 6.0  # 典型室内 WiFi K-factor
    h_los = np.sqrt(K / (K + 1))
    h_nlos = np.sqrt(1 / (2 * (K + 1)))
    h = h_los + h_nlos * (np.random.randn() + 1j * np.random.randn())

    noise = np.random.normal(0, np.sqrt(noise_power), latent.shape)
    received = np.abs(h) * latent + noise
    return received.astype(np.float32)
```

### 7.5 模型-硬件协同优化（Channel Pruning for Cortex-A72）

#### 7.5.1 核心思路

不仅从编译器侧优化执行效率，还从模型侧反向适配硬件特征。对 LGJSCC-c3 的 Generator 做**结构化通道剪枝**，使每层的通道数对齐 NEON 向量化宽度：

- FP32 模式下：通道数对齐到 4 的倍数（NEON float32x4）
- INT8 模式下：通道数对齐到 16 的倍数（NEON int8x16）

#### 7.5.2 为什么这个方向硬核

- "模型-硬件协同设计"（Model-Hardware Co-Design）是当前 AI 芯片领域的研究热点
- 评审会认为团队不仅会用工具优化，还能从模型架构层面做出面向硬件的设计决策
- 结合量化（4.3 节），剪枝后的模型在 INT8 模式下可以获得更好的向量化效率

#### 7.5.3 技术方案

```python
import torch
import torch.nn.utils.prune as prune
import numpy as np

def neon_aware_channel_pruning(model, target_alignment=4, prune_ratio=0.2):
    """
    面向 ARM NEON 的结构化通道剪枝。
    目标：每层输出通道数对齐到 target_alignment 的倍数，
    使 TVM 的 vectorize 变换可以无 padding 地使用 NEON 指令。
    """
    for name, module in model.named_modules():
        if not isinstance(module, (torch.nn.Conv2d, torch.nn.ConvTranspose2d)):
            continue

        out_channels = module.weight.shape[0]

        # 计算每个输出通道的 L1 范数
        channel_norms = torch.norm(
            module.weight.data.view(out_channels, -1), p=1, dim=1
        )

        # 确定剪枝后的目标通道数（对齐到 NEON 向量宽度）
        target_channels = out_channels - int(out_channels * prune_ratio)
        target_channels = max(
            (target_channels // target_alignment) * target_alignment,
            target_alignment  # 至少保留一个向量宽度
        )
        actual_prune_count = out_channels - target_channels

        if actual_prune_count <= 0:
            continue

        # L1 结构化剪枝：移除 L1 范数最小的通道
        prune.ln_structured(
            module, name='weight',
            amount=actual_prune_count / out_channels,
            n=1, dim=0
        )

        print(f"{name}: {out_channels} → {target_channels} channels "
              f"(aligned to {target_alignment})")

    return model

# 使用示例
generator = load_pretrained_generator("lgjscc_c3.pt")
pruned_generator = neon_aware_channel_pruning(
    generator,
    target_alignment=4,   # FP32: 对齐到 4（NEON float32x4）
    prune_ratio=0.2       # 剪掉 20% 的通道
)

# 在原始训练数据上 fine-tune 几个 epoch 恢复精度
optimizer = torch.optim.Adam(pruned_generator.parameters(), lr=1e-4)
for epoch in range(5):
    fine_tune_one_epoch(pruned_generator, train_loader, optimizer)

# 导出剪枝后的 ONNX → TVM 重新编译
torch.onnx.export(pruned_generator, dummy_input, "pruned_decoder.onnx")
```

#### 7.5.4 NEON 向量化效率对比

| 通道数 | NEON FP32 向量化效率 | 说明 |
|---|---|---|
| 48（剪枝前） | 48/4 = 12 次迭代，无尾部 | 效率 100% |
| 40（剪枝后，未对齐） | 40/4 = 10 次迭代，无尾部 | 效率 100%，但通道更少 |
| 36（剪枝后，对齐） | 36/4 = 9 次迭代，无尾部 | 效率 100%，计算量更少 |
| 33（剪枝后，未对齐） | 33/4 = 8 次 + 1 标量尾部 | **效率 ~97%，有尾部开销** |

> 关键洞察：剪枝时如果随意确定目标通道数，可能导致 NEON 向量化出现尾部处理（scalar tail），反而引入额外开销。NEON-aware 剪枝通过强制对齐消除这一问题。

### 11.5 方案设计的核心考量

**（1）为什么 5.1（Profiling）和 7.2（big.LITTLE）放在第一天。**

5.1 是后续所有优化的枢纽——不知道瓶颈在哪，4.2 的调优预算和 7.1 的手写 schedule 都是盲打。7.2 是确定性最高、ROI 最高的硬核方向，且完全不依赖其他任务。两者放在第一天可以同时产出"分析数据"和"优化成果"。

**（2）为什么 4.2（增大预算）放在夜间。**

MetaSchedule 调优是 CPU 密集的长时间任务（2000 trials 预计 6-12 小时），适合在晚上启动、后台运行。Agent 可以 SSH 启动 RPC tracker，在笔记本上启动调优脚本，第二天早上检查结果。关键是基于 5.1 的 profiling 数据，用 `--op-names` 只搜索热点 task，避免预算浪费。

**（3）为什么 4.3（量化）放在最后且标记为"试探性"。**

TVM 0.24dev 的 Relax 量化 API 尚不稳定，是全部任务中风险最高的一项。如果花 2 天调试仍无法跑通，应果断放弃，在文档中写"已尝试量化，当前 TVM 版本的 Relax 量化 API 尚不完善，我们提供了完整的 Relay 路线量化方案设计（见 4.3 节），列为后续工作"。这比花 4-5 天死磕一个不确定的方向要明智得多。

**（4）为什么 7.5、7.3、6.3 被果断放弃。**

- 7.5（剪枝）需要训练数据 + GPU fine-tune，是独立子项目。
- 7.3（自定义 Pass）Relax API 不稳定，调试周期不可控。
- 6.3（动态形状）每个 shape 需独立调优，MNN 已覆盖此需求。

这三项在文档中已有完整的技术设计和代码模板，放弃执行但保留方案设计，评审追问时可以展示技术视野。

**（5）关于 4.3 量化失败的降级策略。**

如果 TVM 量化跑不通，可以用以下替代方案弥补：

- MNN 路线的 FP16/INT8 量化（4.4 节已包含）更容易成功，MNN 的量化工具链成熟度远高于 TVM Relax。
- 在文档中重点强调 MNN 量化结果，将 TVM 路线的量化标注为"后续工作"。
- 评审追问时回答："TVM Relax 的量化 API 在 0.24dev 版本中仍在迭代，我们在 MNN 路线上完成了 INT8 量化并验证了精度可接受性；TVM 量化方案已设计完毕，待后续版本 API 稳定后落地。"

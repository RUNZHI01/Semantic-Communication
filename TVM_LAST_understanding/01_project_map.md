# 项目结构与职责图（TVM_LAST 认知版）

## 1. 目标
本文件用于快速建立“谁负责什么、输入输出在哪里、今晚先做哪块”的统一认知，避免进入代码后迷路。

## 2. 仓库层级（当前会话范围）

| 路径 | 职责 | 今晚是否改动 |
|---|---|---|
| `../TVM_LAST` | 主工程代码（客户端/服务端/TVM集成） | 否（只读理解） |
| `./TVM_LAST_understanding` | 对 `TVM_LAST` 的结构理解、执行链、风险与计划沉淀 | 是 |
| `./session_bootstrap` | 今晚起可持续执行的日志/模板/任务基础设施 | 是 |

> 约束：本阶段只在 `tvm_metaschedule_session_notes` 下写文档与脚手架，不改 `TVM_LAST`。

## 3. 角色划分（按执行责任）

| 角色 | 主要职责 | 关键产物 |
|---|---|---|
| 客户端调用层 | 组织请求参数、触发推理或调优流程、收集结果 | 请求配置、调用日志、结果摘要 |
| 服务端编排层 | 接收请求、校验输入、调度执行（本地或RPC） | 服务日志、任务状态、错误码 |
| TVM 优化层 | Relay/IRModule 生成、Task 抽取、MetaSchedule 调优与应用 | tuning DB、best trace、优化后模块 |
| 结果分析层 | 对比 baseline/current、判断收益和稳定性 | 日报、实验记录、下一轮变量 |

## 4. 数据与产物流向

1. 输入（模型 + shape + target）由客户端层定义并传给服务端。  
2. 服务端将“可执行优化任务”转给 TVM 优化层，产出 tuning 记录与候选最优结果。  
3. 优化后产物回到服务端执行验证，再回传客户端。  
4. 客户端将关键指标写入 `session_bootstrap/reports` 与 `session_bootstrap/logs`。

## 5. 今晚对齐清单（开工前 10 分钟）

1. 明确首批基线：`target`、线程数、固定 shape、测量轮次。  
2. 确认调优模式：`quick`（20-40 分钟）和 `full`（夜间）。  
3. 确认热任务来源：至少拿到前 `3-8` 个热点 task。  
4. 确认日志落盘位置：`session_bootstrap/logs`。  
5. 确认结果模板：使用 `session_bootstrap/templates` 中模板。

## 6. 判定本文件完成的标准

- 新成员 10 分钟内能说清“客户端/服务端/TVM 各自边界”。  
- 今晚执行时不会把产物写到临时目录导致丢失。  
- 可以直接据此进入 `02_execution_flow.md` 执行。
